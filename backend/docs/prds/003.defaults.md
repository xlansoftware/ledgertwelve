# PRD: Default Data Service

## Problem Statement

When a new user registers, a "Main" book is created inline in `AuthController.Register()` but no default categories are seeded. This means every new user gets a book called "Main" with zero categories — they must create categories from scratch before they can meaningfully record transactions. The frontend list of 22 default categories (Groceries, Pets, Maintenance, Utilities, etc.) exists in the frontend codebase but has no server-side equivalent. The inline book creation in the controller also violates Clean Architecture's rule that controllers should contain no business logic.

## Solution

Extract a `DefaultDataService` in the Application layer that idempotently ensures a user has a Main book and a standard set of 22 default categories. The service is called once at registration time, replacing the current inline book creation in `AuthController.Register()`. The service checks whether defaults already exist before creating them, making it safe to invoke in other contexts later (e.g., post-login initialization) without duplicating data.

## User Stories

1. As a new user registering for the first time, I want a Main book created automatically, so that I can start recording transactions immediately.
2. As a new user registering for the first time, I want 22 default categories (Groceries, Pets, Maintenance, Utilities, Dining Out, Transportation, Sport, Entertainment, Miscellaneous, Health / Medical, Personal Care, Clothing, Travel, Gifts, Education, Parents, Insurance, Savings, Taxes, Subscriptions, Rent / Mortgage, Kids) seeded automatically, so that I can categorize transactions without creating categories from scratch.
3. As a returning user who has deleted some categories, I want my remaining categories preserved and not re-seeded, so that my customizations are respected.
4. As a returning user who has deleted their Main book (if that were possible), I want the Main book re-created on next initialization, so that I'm never left without a Main book.
5. As a developer, I want default data creation to live in a dedicated, testable service, so that the registration endpoint remains thin and the logic can be unit-tested independently.

## Implementation Decisions

### Module: `IDefaultDataService` (interface)

- **Location**: `ledger12.Application/Interfaces/IDefaultDataService.cs`
- **Single method**: `Task EnsureDefaultsAsync(Guid userId)`
- The method is idempotent — checks existence before creating, safe to call repeatedly.

### Module: `DefaultDataService` (implementation)

- **Location**: `ledger12.Application/Services/DefaultDataService.cs`
- **Dependencies**: `IBookRepository`, `ICategoryRepository`
- **Constructor injection**: Both repositories are injected directly — the service does not call other services (e.g., `ICategoryService`), avoiding circular dependencies and order-assignment conflicts.
- **Book creation logic**:
  - Call `_bookRepo.GetMainBookAsync(userId)`.
  - If null, create `new Book("Main", userId, "EUR")` and call `_bookRepo.AddAsync(book)`.
- **Category creation logic**:
  - Call `_categoryRepo.GetByUserAsync(userId)`.
  - If the returned list is empty, create each of the 22 default categories with explicit order (1-22) and call `_categoryRepo.AddAsync(cat)` for each.
- **Default categories list**: A `private static readonly` collection of tuples inline in `DefaultDataService`, containing name, recurring flag, hex color, icon name, and display order. This is a deep module — the list is encapsulated within the service and does not leak into any other layer.
- **Atomicity**: Each `AddAsync` call saves individually (matching the existing pattern in all other services). No explicit transaction wrapping. If a failure occurs mid-seed, the registration fails with an HTTP 500 and the user retries — the partial defaults are an acceptable edge case for a registration-time operation.
- **No new database schema changes**: The `Book`, `Category`, and related entities are unchanged. Category IDs are generated via `Guid.NewGuid()` (the existing constructor behavior — no string IDs like `cat_1` are used server-side).

### Module modification: `AuthController`

- **Remove**: `IBookRepository` from constructor injection (no longer needed by the controller).
- **Add**: `IDefaultDataService` to constructor injection.
- **Replace** the inline book creation block in `Register()`:
  ```csharp
  // OLD:
  var userId = Guid.Parse(user.Id);
  var mainBook = new Domain.Entities.Book("Main", userId, "EUR");
  await _bookRepo.AddAsync(mainBook);

  // NEW:
  var userId = Guid.Parse(user.Id);
  await _defaultDataService.EnsureDefaultsAsync(userId);
  ```
- No other controller methods are affected.

### Module modification: `Program.cs`

- **Add** DI registration alongside other application services:
  ```csharp
  builder.Services.AddScoped<IDefaultDataService, DefaultDataService>();
  ```

### Module modification: `DbInitializer`

- **No change**. The `DbInitializer.SeedAsync` seeds a demo user via ASP.NET Identity. That demo user will now get defaults via the same path (the demo user is created with `UserManager`, not via `AuthController.Register()`, so `DbInitializer` won't auto-create defaults for the demo user — this is acceptable since the demo user is for development only, and defaults could be seeded explicitly in `DbInitializer` as a follow-up if needed).

## Testing Decisions

### What makes a good test

- Test external behavior — what the service does, not how it does it.
- Mock all repository dependencies using Moq.
- Verify repository calls were made with the correct parameters.
- Verify repository calls were **not** made when defaults already exist (idempotency).
- For category seeding, verify that all 22 categories are created with correct names, orders, and flags.
- Do not test the Category constructor or Book constructor behavior — those are tested in their own entity tests.

### Module to test

- **`DefaultDataServiceTests`** in `ledger12.Tests/Unit/Application/`.

### Prior art

- Follow the same patterns as `BookServiceTests` in `ledger12.Tests/Unit/Application/`:
  - Constructor sets up mocks: `Mock<IBookRepository>`, `Mock<ICategoryRepository>`.
  - Moq for mocking, `[Fact]` for test methods.
  - Method name format: `MethodName_ExpectedBehaviour_WhenCondition` (e.g., `EnsureDefaultsAsync_CreatesBookAndCategories_WhenNeitherExist`).
  - Use `Verify` to assert repository interactions, `Assert` for DTO values.

### Test scenarios

1. `EnsureDefaultsAsync_CreatesMainBook_WhenNoMainBookExists` — `GetMainBookAsync` returns null, verify `AddAsync` was called with a book named "Main".
2. `EnsureDefaultsAsync_Creates22Categories_WhenNoCategoriesExist` — `GetByUserAsync` returns empty, verify `AddAsync` was called 22 times with categories matching the default names and orders.
3. `EnsureDefaultsAsync_SkipsBookCreation_WhenMainBookExists` — `GetMainBookAsync` returns a book, verify `AddAsync` was NOT called for book.
4. `EnsureDefaultsAsync_SkipsCategoryCreation_WhenCategoriesExist` — `GetByUserAsync` returns non-empty list, verify `AddAsync` was NOT called for categories.
5. `EnsureDefaultsAsync_CreatesCategoriesWithCorrectProperties` — verify each created category has the expected name, recurring flag, color, icon, and sequential order (1-22).

## Out of Scope

- Calling `EnsureDefaultsAsync` on every login — not needed, registration is sufficient.
- Modifying `DbInitializer` to seed defaults for the demo user — can be a separate follow-up if the demo user should also get default categories.
- Adding a database migration — no schema changes required.
- Adding an `IsMain` flag to the `Book` entity — the existing name-based lookup (`b.Name == "Main"`) continues to work.
- Adding default categories as a seed table or configuration file — static data stays in code.
- Any frontend changes — this is purely a backend service.

## Further Notes

- The service is intentionally simple (two repository dependencies, no service-to-service calls). This keeps it testable and avoids coupling between DefaultDataService and other business services (CategoryService, BookService).
- The idempotent design means the service could be called on login in the future if registration-time seeding is not sufficient (e.g., for pre-existing users who registered before this feature existed). Checking `GetByUserAsync(userId)` returning empty covers that migration case transparently.
- The 22 categories mirror what the frontend defines as default categories, ensuring consistency between client and server default sets.
