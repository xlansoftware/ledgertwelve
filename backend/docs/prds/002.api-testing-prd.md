# PRD: ledger12 API Testing

## Problem Statement

A comprehensive backend API for ledger12 is being built from scratch — domain entities, repositories, services, controllers, export/import engines, and exchange rate integration. Without testing, regressions will go undetected, edge cases will be missed, and the financial nature of the application means bugs could lead to data corruption. The existing codebase has no tests.

## Solution

Build a thorough test suite across all layers of the application. Tests follow the project's testing conventions: xUnit + Moq, test class names mirror the class under test, method names follow `MethodName_ExpectedBehaviour_WhenCondition` format, and tests cover only external behavior — not implementation details.

## User Stories

1. As a developer, I want unit tests for all domain entities, so that business rules (status transitions, invariants) are verified in isolation.
2. As a developer, I want unit tests for all application service methods, so that use case logic is verified with mocked dependencies.
3. As a developer, I want unit tests for all FluentValidation validators, so that request validation rules are verified independently.
4. As a developer, I want unit tests for the export file generation logic (CSV, JSON writers), so that exported files match the expected column format and encoding.
5. As a developer, I want unit tests for the import validation engine (preview mode), so that each validation rule is tested without touching a database.
6. As a developer, I want unit tests for the import commit logic (upsert, clearExisting, transaction rollback), so that the import pipeline is correct.
7. As a developer, I want unit tests for DTO mapping extensions, so that entity-to-DTO conversions are correct.
8. As a developer, I want integration tests for key repository queries, so that EF Core LINQ queries produce correct results against a real or in-memory database.
9. As a developer, I want tests for date range boundary conditions (inclusive from, exclusive to), so that filtering is correct at day boundaries.
10. As a developer, I want tests for paginated transaction search with all filter combinations, so that the search endpoint is verified.

## Implementation Decisions

### Test Architecture

- **Test Framework**: xUnit (`ledger12.Tests` project, already configured)
- **Mocking**: Moq — never mock the class under test, only its dependencies
- **Database**: In-memory provider (`Microsoft.EntityFrameworkCore.InMemory`) for repository integration tests
- **Test location**: Unit tests in `ledger12.Tests/Unit/`, integration tests in `ledger12.Tests/Integration/`
- **Convention**: File name = `{ClassName}Tests.cs`, class name = `{ClassName}Tests`, method name = `{MethodName}_{ExpectedBehaviour}_When{Condition}`

### Modules to Test

| Module | Test Type | Priority | Rationale |
|--------|-----------|----------|-----------|
| Domain entities (Book, Transaction, Category, BookShare, ExportJob) | Unit | High | Encapsulate business invariants — must be verified in isolation |
| Application Services (all 8 services) | Unit | High | Core orchestration logic — mocked repos, test each method |
| FluentValidation Validators | Unit | High | Request validation rules are critical for data integrity |
| DTO Mapping Extensions | Unit | Medium | Mechanical but easy to get wrong |
| Export file writers (CSV, JSON) | Unit | Medium | Output format correctness |
| Import validation engine | Unit | High | Complex rules — preview mode must match commit behavior |
| Import commit logic | Unit | High | Transactional behavior, upsert, clearExisting |
| Repositories | Integration | Medium | EF queries need to be tested against a real-ish database |
| AuthController | Unit | Medium | Thin wrapper — mainly tests that services are called correctly |
| ExchangeRateService | Unit | Medium | HTTP call wrapped — mock HttpClient for tests |

### What Makes a Good Test

- Tests **external behavior** — what the method does, not how it does it.
- A test calls a public method with given inputs and asserts on the return value or on mocked dependency interactions.
- No test inspects private state, internal implementation, or intermediate steps.
- Each test covers exactly one scenario (one arrange-act-assert cycle).
- Tests are deterministic — no random data, no time-dependent assertions (use fixed `DateTime` values).
- Mocks are strict about which methods are expected to be called.

### Domain Entity Tests (examples)

```csharp
// BookTests.cs
[Fact]
public void Close_SetsStatusToClosed_WhenBookIsOpen()
{
    var book = new Book("Main", "EUR", ownerId: Guid.NewGuid());
    book.Close();
    Assert.Equal(BookStatus.Closed, book.Status);
    Assert.NotNull(book.ClosedAt);
}

[Fact]
public void Close_ThrowsDomainException_WhenBookIsAlreadyClosed()
{
    var book = new Book("Main", "EUR", ownerId: Guid.NewGuid());
    book.Close();
    Assert.Throws<DomainException>(() => book.Close());
}

[Fact]
public void Reopen_SetsStatusToOpen_WhenBookIsClosed()
{
    var book = new Book("Main", "EUR", ownerId: Guid.NewGuid());
    book.Close();
    book.Reopen();
    Assert.Equal(BookStatus.Open, book.Status);
    Assert.Null(book.ClosedAt);
}

[Fact]
public void Reopen_ThrowsDomainException_WhenBookIsAlreadyOpen()
{
    var book = new Book("Main", "EUR", ownerId: Guid.NewGuid());
    Assert.Throws<DomainException>(() => book.Reopen());
}
```

### Application Service Tests (examples)

```csharp
// BookServiceTests.cs
[Fact]
public async Task GetByIdAsync_ReturnsBookDto_WhenBookExistsAndUserIsOwner()
{
    var bookId = Guid.NewGuid();
    var userId = Guid.NewGuid();
    var book = new Book("Test", "EUR", userId);
    _bookRepo.Setup(r => r.GetByIdAsync(bookId)).ReturnsAsync(book);
    _currentUser.Setup(c => c.UserId).Returns(userId);

    var result = await _service.GetByIdAsync(bookId);

    Assert.NotNull(result);
    Assert.Equal("Test", result.Name);
}

[Fact]
public async Task GetByIdAsync_ThrowsNotFoundException_WhenBookDoesNotExist()
{
    var bookId = Guid.NewGuid();
    _bookRepo.Setup(r => r.GetByIdAsync(bookId)).ReturnsAsync((Book?)null);

    await Assert.ThrowsAsync<NotFoundException>(() => _service.GetByIdAsync(bookId));
}

[Fact]
public async Task GetByIdAsync_ThrowsNotFoundException_WhenUserIsNotOwnerOrShared()
{
    var bookId = Guid.NewGuid();
    var userId = Guid.NewGuid();
    var otherUserId = Guid.NewGuid();
    var book = new Book("Test", "EUR", otherUserId);
    _bookRepo.Setup(r => r.GetByIdAsync(bookId)).ReturnsAsync(book);
    _currentUser.Setup(c => c.UserId).Returns(userId);

    await Assert.ThrowsAsync<NotFoundException>(() => _service.GetByIdAsync(bookId));
}
```

## Prior Art

- The Tests project already references xUnit, Moq, and EF Core InMemory.
- The Tests project references `ledger12.Application`, `ledger12.Domain`, and `ledger12.Infrastructure`.
- Test method naming convention: `MethodName_ExpectedBehaviour_WhenCondition` (documented in AGENTS.md).
- No existing tests to follow — these will be the first tests in the project.

## Out of Scope

- End-to-end / integration tests against the full HTTP stack (no `WebApplicationFactory` or `TestServer` tests for v1).
- Snapshot tests for export files.
- Performance / load tests.
- UI / frontend tests.
- Property-based testing (F#s Check, etc.).
- Tests for the DbInitializer seed logic.
- Tests for ASP.NET Identity internals (cookie generation, password hashing).

## Further Notes

- Tests are written **alongside** or **immediately after** the implementation of each module — not deferred to the end.
- Each service test class uses a shared pattern: a `Setup` method that creates fresh mock instances of all dependencies and a fresh service instance for each test.
- Repository integration tests use a fresh in-memory database per test class to avoid test pollution.
- Any test that touches `DateTime.UtcNow` uses a fixed date via an `IClock` abstraction or a static constant.
- Tests that verify exception behavior use `Assert.ThrowsAsync<T>` for async methods, `Assert.Throws<T>` for sync methods.
- Validator tests use FluentValidation's `TestValidate` extension for cleaner assertion syntax.
