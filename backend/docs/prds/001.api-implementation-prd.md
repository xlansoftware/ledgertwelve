# PRD: ledger12 API Implementation

## Problem Statement

The backend for ledger12 is a scaffolded ASP.NET Core project with Identity wired up but no domain model, no business logic, no repositories, no services, and only a stub `AuthController`. The project references interfaces and services that don't exist yet and does not compile. A comprehensive API contract (`API.md`) has been written covering authentication, categories, books, transactions, sharing, reporting, exchange rates, exports, and imports — but none of this has been implemented.

## Solution

Build out the full backend following Clean Architecture and the `API.md` contract. Create domain entities, repository interfaces and implementations, service layer with focused service interfaces, API controllers that delegate to services, and all supporting infrastructure (EF Core configuration, migrations, middleware, DTOs, validators, background export processing, import engine, exchange rate integration).

## User Stories

1. As a user, I want to register with email and password, so that I can create an account and start using the ledger.
2. As a registered user, I want to log in with email and password, so that I can access my financial data securely.
3. As a logged-in user, I want to log out, so that my session is terminated.
4. As a logged-in user, I want to see who I am (whoami), so that I can verify my authentication state.
5. As a user, I want a Main book created automatically when I register, so that I always have a book to record transactions in.
6. As a user, I want to create new books with a name and currency, so that I can organize transactions by context (e.g., vacation, project).
7. As a user, I want to list all books I own or have been shared into, so that I can navigate between them.
8. As a user, I want to view a single book's details including sharing info, so that I can see who has access.
9. As a user, I want to update a book's name and currency, so that I can correct or change metadata.
10. As a user, I want to delete an empty book (not the Main book), so that I can clean up unused books.
11. As a user, I want to get and set my currently selected book, so that the UI remembers which book I'm working in across sessions.
12. As a user, I want to close a book by creating a balancing transaction in Main, so that I can finalize a book's finances.
13. As a user, I want to reopen a closed book, so that I can continue using it if needed.
14. As a user, I want to view book statistics (transaction count, net sum) optionally filtered by date, so that I can get a quick overview.
15. As a user, I want to create categories with name, color, icon, and recurring flag, so that I can classify my transactions.
16. As a user, I want to list all my categories, so that I can select from them when creating transactions.
17. As a user, I want to update a category's properties, so that I can rename or recolor it.
18. As a user, I want to delete a category optionally reassigning its transactions to another category, so that I can clean up my category list without losing data.
19. As a user, I want to bulk-reassign transactions from one category to another, so that I can correct misclassified transactions.
20. As a user, I want to reorder my categories by sending a full ordered list of category IDs, so that they display in my preferred order.
21. As a user, I want to create transactions with date, amount, optional multi-currency fields, category, and note, so that I can record my financial activity.
22. As a user, I want to list/search transactions with filters (book, date range, category, creator, note text, amount range, pagination), so that I can find specific transactions.
23. As a user, I want to view a single transaction's full details, so that I can inspect or edit it.
24. As a user, I want to update a transaction's fields, so that I can correct mistakes.
25. As a user, I want to delete a transaction, so that I can remove erroneous entries.
26. As a user, I want to share my book with another user with edit or view permission, so that we can collaborate on a shared ledger.
27. As a user, I want to change a shared user's permission on my book, so that I can grant or restrict access.
28. As a user, I want to remove a user's access to my book, so that I can revoke sharing.
29. As a user, I want to globally share all my books with another user (edit permission), so that I don't need to share each book individually.
30. As a user, I want to remove a global share, so that the user loses access to all my books.
31. As a user, I want to look up an exchange rate between two currencies, so that I can fill in multi-currency transaction fields.
32. As a user, I want to see a list of all users I've interacted with (myself + book collaborators), so that I can identify transaction authors.
33. As a user, I want to view a totals report grouped by day/week/month/year over a date range, so that I can see income vs expense trends.
34. As a user, I want to view a category breakdown report over a date range, so that I can see where my money is going.
35. As a user, I want to view daily net amounts over a date range, so that I can see day-by-day balance changes.
36. As a user, I want to view monthly net amounts over a date range, so that I can see month-by-month balance changes.
37. As a user, I want to view the average daily net amount over a date range, so that I can compute projection rates for charts.
38. As a user, I want to view the average monthly net amount over a date range, so that I can compute projection rates for charts.
39. As a user, I want to create an async export job for categories, transactions, books, reports, or a full backup in CSV, XLSX, or JSON format, so that I can download my data.
40. As a user, I want to check the status of an export job, so that I can poll for completion.
41. As a user, I want to download a completed export file, so that I can save it locally.
42. As a user, I want to import data (transactions, categories, books) with a preview mode, so that I can see what will change before committing.

## Implementation Decisions

### Architecture

- **Clean Architecture** with layers: API → Infrastructure → Application → Domain. No inward dependency violations.
- **Service layer** pattern — not CQRS/MediatR. Focused service interfaces per domain area replace the monolithic `ILedgerService`/`IDashboardService` which are removed.
- Every endpoint returns typed `ActionResult<T>` — never `IActionResult` alone.

### Domain Layer

- **Entities** (all in `ledger12.Domain`):
  - `Book` — `Id` (Guid), `Name`, `Currency?`, `Status` (Open/Closed), `OwnerId` (Guid), `CreatedAt`, `ClosedAt?`, navigation to `BookShare`s
  - `Category` — `Id`, `Name`, `Recurring` (default false), `Color?`, `Icon?`, `Order` (int), `UserId` (Guid), `CreatedAt`
  - `Transaction` — `Id`, `BookId`, `UserId`, `DateTime`, `Amount` (decimal), `OriginalCurrency?`, `OriginalAmount?`, `ExchangeRate?`, `CategoryName?` (denormalized string — not an FK), `Note?`, `CreatedAt`, `IsBookClosingEntry` (default false), `ClosedBookId?`
  - `BookShare` — `BookId` (Guid), `UserId` (Guid), `Permission` (View/Edit), navigation to `Book`
  - `GlobalShare` — `OwnerId` (Guid), `SharedWithUserId` (Guid), `CreatedAt`
  - `UserPreference` — `UserId` (Guid, PK), `CurrentBookId?` (Guid)
  - `ExportJob` — `Id` (Guid), `Status` (Pending/Processing/Completed/Failed), `Format` (Csv/Xlsx/Json), `ContentType` (enum), `BookId?`, `UserId`, `FilePath?`, `ErrorMessage?`, `CreatedAt`
  - `CurrencyRate` — `FromCurrency` (string), `ToCurrency` (string), `Rate` (decimal), `UpdatedAt`
- **No value objects for v1** — plain properties on entities.
- **Enums**: `BookStatus`, `BookPermission`, `ExportJobStatus`, `ExportFormat`, `ExportContentType`
- `Id` for all entities is `Guid` — stored as GUID in the DB. No prefixes (`book_`, `tx_`, etc.) used internally. The API.md examples use short readable IDs; a note is added to API.md clarifying that all actual IDs are GUIDs.

### Application Layer

- **Service interfaces** (in `ledger12.Application.Interfaces`):
  - `ICurrentUserService` — `UserId` (Guid property, extracted from `HttpContext`)
  - `IBookService` — CRUD + close/reopen + stats + current-book get/set + sharing
  - `ICategoryService` — CRUD + reassign + reorder
  - `ITransactionService` — CRUD + search with pagination and filters
  - `IReportService` — totals, categories, daily, monthly, average/daily, average/monthly
  - `IExchangeRateService` — `GetExchangeRateAsync(from, to)` returning `decimal?`
  - `IExportService` — create job, get status, get download URL
  - `IImportService` — preview + commit import
  - `IUserService` — whoami, list users

- **Repository interfaces** (also in `ledger12.Application.Interfaces`, implementations in Infrastructure):
  - `IBookRepository`, `ICategoryRepository`, `ITransactionRepository`, `IReportRepository`

- **DTOs** (records in `ledger12.Application.DTOs`): `LoginRequest`, `RegisterRequest`, `UserSummary`, `CategoryDto`, `BookDto`, `SharedUserDto`, `TransactionDto`, `CreateTransactionRequest`, `UpdateTransactionRequest`, `CreateCategoryRequest`, `UpdateCategoryRequest`, `CreateBookRequest`, `UpdateBookRequest`, `CloseBookRequest`, `SetCurrentBookRequest`, `AddShareRequest`, `UpdateShareRequest`, `TotalsReportItem`, `CategoryReportItem`, `DailyReportItem`, `MonthlyReportItem`, `AverageReportItem`, `ExchangeRateResponse`, `ExportResponse`, `ExportStatusResponse`, `ImportRequest`, `ImportResponse`, `BookStatsResponse`, `DeleteCategoryResponse`, `ReassignCategoryResponse`, `ReorderResponse`, `DeleteTransactionResponse`, `CloseBookResponse`, `ReopenBookResponse`, `CreateShareResponse`, pagination `Meta` type, plus the standardized error response shape `{ error: string }`.

- **Mappings** (extension methods in `ledger12.Application.Mappings`): One file per entity type with `ToDto()` extensions. Explicit, no AutoMapper.

- **Validators** (FluentValidation in `ledger12.Application.Validators`): One validator per unique request shape.

### API Layer

- **Route prefix**: All endpoints under `/api/v1`
- **Controllers**: `AuthController`, `BooksController`, `CategoriesController`, `TransactionsController`, `SharesController`, `RatesController`, `ReportsController`, `ExportsController`, `ImportsController`
- **Auth**: Cookie-based (HTTP-only, Secure, SameSite=Lax). `[Authorize]` on all controllers except login, register, whoami, and rates/exchange.
- **Registration**: Custom `POST /api/v1/auth/register` added to API.md — `{ email, password }` creates the user + Main book atomically.
- **Current user**: `ICurrentUserService` implemented via `IHttpContextAccessor` extracting `ClaimTypes.NameIdentifier`.
- **Error handling**: `ExceptionMiddleware` catches `NotFoundException` → 404 `{ error }`, `DomainException` → 400 `{ error }`, validation failures → 400 `{ error }`, unhandled → 500 `{ error }`.
- **Validation integration**: FluentValidation auto-validation (`AddFluentValidationAutoValidation()`) with a custom `InvalidModelStateResponseFactory` that flattens errors to `{ error: "summary message" }`.
- **JSON serialization**: `System.Text.Json` with `JsonStringEnumConverter` for enum DTOs.

### Infrastructure Layer

- **Database**: SQLite via EF Core. Single `AppDbContext` with `DbSet<>` for each entity.
- **Migrations**: One `InitialCreate` migration containing the full schema.
- **EF Configuration**: Entity configurations via `IEntityTypeConfiguration<T>` in a `Configurations/` folder.
- **Repositories**: EF-based implementations of all repository interfaces.
- **Exchange rates**: `HttpClient` consuming Frankfurter API (`https://api.frankfurter.app`). No API key required for basic use; configurable via `AppConfig:EXCHANGE_RATE_API_KEY` for future provider swap.
- **Export storage**: Local file system in a configured directory. Files named per API.md conventions (e.g., `transactions-Main-2026-06-24.xlsx`).
- **Export processing**: `Channel<ExportJob>` with `ExportJobProcessor` (BackgroundService) consuming jobs. CSV/JSON generation via `System.Text.Json` + string building. XLSX via a lightweight library or structured CSV-as-XLSX approach.
- **Seed**: `DbInitializer` updated to create demo user, Main book, demo categories, deterministic transactions for 2025-2026.
- **Global share on book creation**: `IBookService.CreateAsync` queries `GlobalShare` table for the owner and creates `BookShare` entries for each.

## Out of Scope

- Password reset, email verification, and other Identity account management flows (handled by ASP.NET Identity UI defaults).
- `view` permission for global shares (listed as future follow-up in API.md).
- Real-time notifications when exports complete (polling-based instead).
- External file storage (S3/Azure Blob) for exports — local filesystem only.
- Rate limiting, API versioning beyond `/api/v1`.
- Admin endpoints, user management UI.
- WebSocket or SignalR for live updates.
- Database providers other than SQLite.
- Caching layer for reports or exchange rates.

## Further Notes

- All IDs are plain GUIDs stored in the database and serialized as standard GUID strings in JSON. No prefixed IDs are used anywhere in the codebase. The API.md examples use shortened IDs for readability only.
- Import is partial-success by design: valid rows commit, failed rows are reported.
- Export is asynchronous: jobs are queued via `Channel<ExportJob>` and processed by a `BackgroundService`. The client polls `GET /exports/{jobId}` for completion.
- The Main book is created at registration time via an Identity event hook, not in the DbInitializer alone.
- The DbInitializer seeds a demo user (`demo@example.com` / `Example-1`) with a Main book, 22 categories, and deterministic transactions for 2025-2026.
- Exchange rates are fetched live from the Frankfurter API — no cached/static table for v1.
- All date ranges follow the convention: `from` inclusive, `to` exclusive.
