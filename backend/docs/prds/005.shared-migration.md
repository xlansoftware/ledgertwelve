# 005 — Shared Book & Global Share Migration

## Problem Statement

When users shared Spaces in ledger11, the current migration (`004.migration.md`) creates an individual Book per Space with per-Space BookShares. This results in:

- Each user gets their **own** separate "Main" book at first login (via `EnsureDefaultsAsync`), even when they shared everything in ledger11.
- There is no concept of a **shared Main book** — both users have their own private Main with no overlap.
- There are no **GlobalShare** records, so sharing must be managed per-book (which was not the old UX pattern).
- The old "Ledger" space (the user's primary book in ledger11) becomes just another book named "Ledger" — its transactions are disconnected from the Main book concept in ledger12.

This means a migrated user who shared all their finances with a partner in ledger11 will find those books are scattered across separate Main books, owned by different users, with no automatic visibility. The ledger12 GlobalShare feature was designed exactly for this pattern but is not used during migration.

## Solution

Modify the migration tool (`ledger12.MigrateFromLedger11`) so that:

1. Spaces named **"Ledger"** (case-insensitive) are **not** converted to separate Books. Instead, their transactions are routed into the **"Main"** book, which is created during migration.
2. When two or more users shared at least one Space in ledger11, they are treated as a **collaboration group** in ledger12 — they share a single Main book and get reciprocal GlobalShare access to all each other's books.
3. Single users with a "Ledger" space get their own private `Book("Main", ...)` created at migration time.
4. **Reciprocal GlobalShare records** are created between all group members (A shares with B, B shares with A).
5. **BookShare records** are derived from each GlobalShare, granting `edit` access to every book each owner owns.
6. All group members' `UserPreference.CurrentBookId` points to the shared Main.
7. The `GetMainBookAsync` repository method is updated so that `EnsureDefaultsAsync` finds an existing Main book without creating a duplicate.

Per-Space books (other than "Ledger") remain individually owned — their `OwnerId` is unchanged. The GlobalShare mechanism gives all group members mutual edit access to every book each of them owns, matching the ledger11 collaboration model.

## User Stories

1. As a migrated user who shared a Space in ledger11, I want to see the **same Main book** as the other user, so that we both start on the same home page.
2. As a migrated user who shared a Space in ledger11, I want the other user's books to appear automatically in my book list, so that I don't need to request per-book sharing.
3. As a migrated user who shared a Space in ledger11, I want the same for my own books — the other user should see them automatically, so that collaboration works both ways.
4. As a migrated user who had a "Ledger" space, I want its transactions to appear in the **Main book** in ledger12, not in a separate "Ledger" book, so that the Main book functions as my primary ledger.
5. As a migrated user who did NOT share any Spaces with anyone and had no "Ledger" space, I want to still get my own private Main book at first login (as before), so that single-user migration is unchanged.
6. As a migrated single user who had a "Ledger" space, I want my Main book created during migration with my "Ledger" transactions already in it, so that I don't see an empty Main book at first login.
7. As a migrated user in a collaboration group, I want to have `edit` permission on all the other user's books (not just the shared Spaces), so that I can manage finances together fully.
8. As the migration operator, I want to see a count of GlobalShare records created in the summary, so that I can verify the collaboration structure was set up correctly.
9. As a user who logs in after migration, I want `EnsureDefaultsAsync` to detect the already-existing shared Main book and skip creating a new one, so that no duplicate Main books are created.
10. As a user in a 3+ member group (e.g., A shares with B, A shares with C), I want all users to share the same Main book and all see each other's books, so that the connected group acts as a single collaboration unit.

### Migration Phases (Revised)

| Phase | Purpose | Key Logic |
|-------|---------|-----------|
| **1** | Users | Unchanged from 004. Create `AppUser` records from old `AspNetUsers`, build email→GUID and old-user-ID→new-user-ID mappings. |
| **2a** | Books from non-"Ledger" Spaces | **Updated.** Convert each Space to a Book, **except** spaces named "Ledger" (case-insensitive comparison). Determine `OwnerId` via `CreatedByUserId` or first migrated member. |
| **2b** | Main Books from "Ledger" Spaces | **New.** For each "Ledger" space: create a `Book("Main", ownerId, "EUR")`. `ownerId` is determined by the connected component logic (see below). Register the space's `SpaceFileId` → Main book GUID so Phase 5 routes transactions to the Main book instead. |
| **3** | GlobalShares + BookShares | **Replaces old Phase 3.** For each connected component with 2+ users: (a) create reciprocal `GlobalShare` records between all pairs of users, (b) for each GlobalShare, create `BookShare(bookId, sharedWithUserId, Edit)` on every book owned by the sharer (including the shared Main). |
| **4** | Categories | Unchanged from 004. Categories from "Ledger" spaces are still migrated (they're per-user, not per-book). |
| **5** | Transactions | **Updated.** When processing a "Ledger" space, the transactions are assigned to the Main book (resolved via `SpaceFileIdToMainBookId` map) instead of a per-Space book. Non-Ledger spaces behave as before. |
| **6** | User Preferences | **Updated.** For users in a connected component, point `CurrentBookId` to the **shared Main**. For single "Ledger" users, point to their own Main. For other single users, keep existing behavior (old `CurrentSpaceId` → book GUID, or null). |

### Connected Component & Main Book Ownership Algorithm

Two users are considered connected if they share **at least one Space** (i.e., both are members of the same Space). 

1. For each Space with migrated members, collect all migrated member user IDs.
2. Build an adjacency list: for each user, list all other users they share a Space with.
3. Compute connected components via BFS/DFS.
4. For each component:
   - **Component size ≥ 2**: Find the user with the oldest `CreatedAt` among all Spaces they belong to. This user becomes the **shared Main book owner**. Create one shared `Book("Main", ownerId, "EUR")`.
   - **Component size == 1** and user has a **"Ledger" space**: Create a private `Book("Main", userId, "EUR")` owned by that user.
   - **Component size == 1** and user has **no "Ledger" space**: No Main book created — left to `EnsureDefaultsAsync` at first login.
5. For each "Ledger" space, register `SpaceFileId → MainBookId` in the context so Phase 5 can route transactions.

### "Ledger" Space Detection

A space is considered a "Ledger" space (and its book creation is skipped, its transactions routed to Main) when:
- `string.Equals(space.Name, "Ledger", StringComparison.OrdinalIgnoreCase)` is true.

This matches "Ledger", "ledger", "LEDGER", etc.

### GlobalShare Creation

For each component with 2+ users `{U1, U2, ..., Un}`:

```
for each pair (Ui, Uj) where i != j:
    create GlobalShare(ownerId: Ui, sharedWithUserId: Uj)
    for each Book where Book.OwnerId == Ui:
        create BookShare(bookId: Book.Id, userId: Uj, permission: Edit)
```

This mirrors the runtime behavior of `BookService.AddGlobalShareAsync`, which creates explicit `BookShare` records for all current books when a global share is added. Note that books from Phase 2a (non-Ledger spaces) plus the shared Main from Phase 2b are all included.

### `GetMainBookAsync` Contract Change

The method signature remains `Task<Book?> GetMainBookAsync(Guid userId)`, but the semantics change:

- **Before:** Returns the "Main" book **owned by** the user, or falls back to the user's first open book.
- **After:** Returns any "Main" book **visible to** the user (owned or shared), or falls back to the user's own first open book.

This ensures that when `EnsureDefaultsAsync(userId)` runs for a migrated user in a group, it finds the shared Main and does not create a duplicate.

### Unchanged Behavior

- Single users without a "Ledger" space still get their Main lazily at first login — no migration-time change.
- Per-Space book ownership (`OwnerId`) is unchanged for non-Ledger spaces.
- Categories migration is identical to 004.migration.md (categories are per-user, not per-book).
- The migration still runs inside a single database transaction (all-or-nothing).

## Testing Decisions

This is a one-time migration tool — no unit or integration tests will be written for the modified modules. The rationale matches 004.migration.md:

- The tool runs once against a copy of production data.
- The all-or-nothing transaction boundary protects against partial failure.
- Manual verification of the final data covers correctness (check GlobalShares table, BookShares table, shared Main book, transaction counts in Main vs other books).
- Writing tests for a one-time tool would have negative ROI.

## Out of Scope

- Three-way or N-way conflict resolution if two users both claim earliest Space creation tie — not expected in practice; deterministic sort by GUID breaks ties.
- Per-space category deduplication changes — no new logic needed, already handled in Phase 4.
- UI changes — the API `POST /api/v1/shares` endpoint already exists and correctly creates GlobalShares. No API changes are needed.
- Changes to `EnsureDefaultsAsync` itself — the updated `GetMainBookAsync` is sufficient to prevent duplicate Main creation.
- Changes to `DefaultDataService` — single-user path (no Ledger space) is unchanged.
- Changes to any API controller or service beyond `BookRepository.GetMainBookAsync`.
- Tests for the migration tool.
- Routing of non-"Ledger" space transactions to Main — only "Ledger" spaces get this treatment.

## Further Notes

- This PRD extends 004.migration.md. The modules, architecture, and phase numbering from 004 are preserved where unchanged.
- The tool writes directly to `AppDbContext.DbSet<>` as before — the repository layer is bypassed for bulk operations.
- The tool uses `Microsoft.Data.Sqlite` for reading legacy databases (unchanged).
- After migration, `DbInitializer.SeedAsync` should NOT be called (real users exist, no demo user needed).
- The Main book's currency is hardcoded to `"EUR"`, matching the `DefaultDataService` default. This is the safest default since we cannot determine intent from the old data.
- A "Ledger" space is still included in the connected component graph (its membership still creates links between users), even though no Book is created for it.
