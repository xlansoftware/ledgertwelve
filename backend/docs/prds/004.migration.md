# 004 — Migrate from Ledger11

## Problem Statement

The ledger12 application needs to be populated with existing user data from four legacy ledger11 SQLite databases. Six users exist in the old `appdata.db` with accounts, password hashes, and preferences spread across 10 space-specific database files. Manually recreating this data in ledger12 is impractical — there are 1,200+ transactions, 100+ categories, and complex sharing relationships across multiple "Spaces" (the old concept of "Books").

The old and new data models differ significantly:
- Old "Spaces" + "SpaceMembers" → new "Books" + "BookShares"
- Per-space categories (scoped to a space) → per-user categories (scoped to the user)
- Transaction `User` stored as email string → new `UserId` GUID foreign key
- Old transaction `Value` as TEXT → new `Amount` as decimal
- Old categories/transactions use auto-increment int IDs → new uses GUIDs throughout
- Old has `TransactionDetail` sub-table with no equivalent in the new model

## Solution

A one-time CLI migration tool (`ledger12.MigrateFromLedger11`) that:

1. Drops the existing ledger12 database and applies fresh migrations.
2. Reads data from the legacy `appdata.db` and matching `space-{guid}.db` files.
3. Migrates users while preserving their existing ASP.NET Core Identity password hashes.
4. Converts old Spaces to new Books, preserving the original GUIDs.
5. Converts SpaceMembers to BookShares with Edit permission.
6. Migrates categories from per-space to per-user, deduplicating by name.
7. Migrates all transactions, resolving email-based user references to GUIDs and category int IDs to category name strings.
8. Sets user preferences (`CurrentBookId`) based on their old `CurrentSpaceId`.

The tool runs once, produces a populated ledger12 database, and is then discarded.

## User Stories

1. As an existing ledger11 user A, I want my account, password, and all my data (books, transactions) migrated so that I can log in to ledger12 without interruption.
2. As an existing ledger11 user B, I want my shared book memberships and permissions preserved so that I can access the same shared books in ledger12.
5. As a migrated user, I want to be placed in the same book I was last using, so that I don't lose my place.
6. As a migrated user who shared a book, I want the categories from that book to be available to me in ledger12.
7. As a migrated user, I want transactions authored by unknown/deleted users to be attributed to the book owner so that the data is not lost.
8. As the migration operator, I want the tool to start with a fresh database so that there are no conflicts between old and new data.
9. As the migration operator, I want a clean summary of what was migrated (users, books, categories, transactions) so that I can verify the result.

## Implementation Decisions

### Module Architecture

The tool is decomposed into four modules:

1. **`OldDatabaseReader`** — Responsible for all SQLite I/O against the legacy databases. Reads `appdata.db` and each matching `space-{guid}.db` file. Provides typed queries: users, spaces, space members, categories (per space), transactions (per space). Encapsulates the SQLite implementation detail behind a clean interface. Consumes no types from the new ledger12 domain — returns only old-data-shaped records.

2. **`DataMapper`** — The core transformation module. Converts old data shapes to new ledger12 entities. Stateless — takes old records plus a `MigrationContext` and returns new entities. Handles all the mapping rules:
   - Email string → user GUID (looked up in context)
   - Old int CategoryId → category name string
   - Old TEXT `Value` → decimal `Amount`
   - Old TEXT `Date` → `DateTimeOffset`
   - Legacy `AccessLevel` → `BookPermission.Edit` (all members get edit)
   - Category deduplication: same-name categories across different spaces become a single per-user category (first color/icon wins)
   - Negatives are valid amounts (they represent withdrawals; no special handling needed)

3. **`MigrationContext`** — In-memory state holding mapping registries built during migration:
   - Old email → new user GUID (populated after user migration)
   - Old Space GUID → new Book GUID (populated after book migration)
   - Old space file ID → new Book GUID (for resolving transactions)
   - Per-user category name → deduplicated (tracked to avoid duplicates)
   - Old category int ID → name string (per space, for resolving transactions)

4. **`MigrationEngine`** — Coordinates the migration in strict phase order. Runs inside a single database transaction (all-or-nothing). Writes through the existing `AppDbContext` directly (not through repositories — this is a bulk operation, the repository abstraction adds overhead without value for a one-time tool).

### Migration Phases (in order)

1. **Users** — Insert all 4 users into `AspNetUsers`. Copy `Id`, `UserName`, `NormalizedUserName`, `Email`, `NormalizedEmail`, `EmailConfirmed`, `PasswordHash`, `SecurityStamp`, `PhoneNumber`, `LockoutEnabled`, `AccessFailedCount` directly. Generate new `ConcurrencyStamp`. Populate `MigrationContext` with email→GUID mappings.

2. **Books** — Convert 4 Spaces to Books. Preserve the old Space GUID as the new Book GUID. Map `Space.Name` → `Book.Name`, `Space.Currency` → `Book.Currency`. Set `Book.OwnerId` to the resolved owner GUID. For orphan space A99DA176 (no valid creator), assign the first SpaceMember (skkcam) as owner. Set `Book.Status` = Open, `Book.CreatedAt` = old `Space.CreatedAt`.

3. **Book Shares** — Convert SpaceMembers to BookShares. All members get `BookPermission.Edit`. Skip members whose users were not migrated (skkdevcraft). Assign permissions only for the members the user specified:
   - A99DA176 (Ledger): skkcam + d.gencheva only
   - D4B92833 (Greece 2025): skkcam + d.gencheva
   - 9C86536D (Море 2025): skkcam + d.gencheva
   - B44330DB (Ledger): tchitera93 only (single-user, no shares needed for owner)

4. **Categories** — For each space, read its `Categories` table. For each member of that space, create categories in the new database with the member's `UserId`. Deduplicate by category name per user (if a user already has a "Groceries" category from a previous space, reuse the existing Category record; first color/icon wins). Map `DisplayOrder` → `Order`, set `Recurring` = false, `CreatedAt` = migration timestamp. Populate `MigrationContext` with per-space int CategoryId → name resolution table.

5. **Transactions** — For each space, read its `Transactions` table. For each transaction:
   - Parse `Value` (TEXT) as decimal → `Amount`
   - Parse `Date` (TEXT) as DateTimeOffset → `DateTime`
   - Resolve `User` (email) via MigrationContext → `UserId` GUID. If email is unknown or empty, assign the book owner's GUID.
   - Resolve `CategoryId` (int) via MigrationContext → `CategoryName` (string). If null, leave as null.
   - Copy `Notes` → `Note`
   - Leave multi-currency fields (`OriginalCurrency`, `OriginalAmount`, `ExchangeRate`) as null
   - Set `IsBookClosingEntry` = false, `ClosedBookId` = null
   - Generate new GUID for `Id`

6. **User Preferences** — For each user, create a `UserPreference` record. If the user had a `CurrentSpaceId` in the old AspNetUsers, resolve it to a Book GUID and set `CurrentBookId`. If the space doesn't map to any migrated book, leave `CurrentBookId` as null.

### What Gets Skipped

- **User accounts**: with no password hashes, no substantive data
- **TransactionDetail table**: Only 6 records across all spaces, redundant itemization
- **Orphan space files**: 4 space-*.db files with no corresponding `Spaces` row in appdata.db (abandoned/legacy data)
- **Dead space**: No owner, no members, zero transactions
- **Widgets table**: No equivalent in new model
- **ApiKeys table**: No equivalent in new model
- **Roles/RoleClaims**: No roles used in old data
- **__EFMigrationsHistory, __EFMigrationsLock, _litestream_***: EF Core and Litestream internal tables

### GUID Preservation

Old GUIDs from ledger11 are reused as the new GUIDs wherever they exist:
- `AspNetUsers.Id` → `AppUser.Id`
- `Spaces.Id` → `Book.Id`
- New GUIDs are generated for entities that used auto-increment int IDs (Categories, Transactions)

| Entity | Count |
|---|---|

## Testing Decisions

No tests will be written for this tool. It is a one-time migration utility that will be discarded after use. The risk of a faulty migration is mitigated by: (a) running against a copy of the production database, (b) the all-or-nothing transaction boundary, and (c) manual verification of the final data.

## Out of Scope

- Recurring/reusable import of ledger11 format data through the API (this is a one-time tool, not a general-purpose importer)
- Password migration for users without password hashes (skipped entirely)
- Migration of the `TransactionDetail` table
- Migration of orphan space files with no matching `Spaces` record
- Migration of dead spaces with no owner or members
- Support for ledger11's widget system
- Any form of incremental/rolling migration
- UI in the ledger12 app for triggering this migration
- Test coverage for the migration tool

## Further Notes

- The tool references the existing `ledger12.Infrastructure` project for `AppDbContext` and existing entity configurations. It does NOT use the repository layer — it writes directly to `DbSet<>` for performance (bulk insert of 1,000+ records).
- The tool should use `Microsoft.Data.Sqlite` for reading the old databases (not EF Core — we don't want EF Core models for the old schema).
- The tool must be run from the solution root so it can resolve the ledger11data folder path relative to the working directory.
- After migration, the existing `DbInitializer.SeedAsync` should NOT be called (the demo user is not needed when real users exist).
- The database connection string should be read from the API project's `appsettings.json` or overridden via CLI argument.
