# PRD 007 — Daily Insight Page

## Problem Statement

The existing insight page (`/insight`) shows a monthly expense breakdown by category — a single pie chart covering the current calendar month. Users cannot see:

- How their spending evolved day by day
- Whether spending is accelerating or slowing across recent days
- A daily breakdown of expenses and income by category for a specific day
- A trend projection showing where their accumulated spending is heading

The monthly view provides a static snapshot but no temporal awareness. Users need to understand their spending trajectory over recent days to answer questions like "Am I on track this week?" and "Which days are driving my expenses?"

## Solution

Introduce a **Daily Insight** page (`/insight/daily`) with three coordinated sections:

1. **Pie chart** — shows expense and income breakdown by category. Defaults to today; switches to any selected day from the list below.
2. **Area chart** — shows accumulated daily spending across the last 10 days with a dotted trend-line projection for the next 10 days, so users can see where their spending trajectory is heading.
3. **Selectable daily list** — shows the net amount for each of the last 10 days. Tapping a day switches the pie chart to show that day's category breakdown.

Together, these three sections let users explore their daily spending patterns, drill into specific days, and see a forward projection — all on one page.

## User Stories

1. As a user, I want to see today's expenses broken down by category in a pie chart, so that I can quickly understand where my money went today.
2. As a user, I want to see today's income broken down by category as a secondary ring in the same pie chart, so that I can compare income vs. expenses at a glance.
3. As a user, I want to see an accumulated area chart of my daily spending over the last 10 days, so that I can see how my total spending has grown day by day.
4. As a user, I want the area chart's Y axis to auto-scale based on the data, so that even small changes in spending are visible.
5. As a user, I want a dotted trend line projecting the next 10 days based on my average daily spending rate, so that I can anticipate where my monthly spending is heading.
6. As a user, I want the trend line to be visually distinct (different color, dotted) from the actual data line, so that I don't confuse projections with real spending.
7. As a user, I want to hover on any point in the area chart and see both the daily amount and the accumulated total for that day, so that I get precise values.
8. As a user hovering on a trend-line point, I want the tooltip to clearly label it as "Projected," so that I know it's a forecast.
9. As a user, I want the area chart to handle days when income exceeds expenses (crossing below zero), so that the chart accurately reflects my net position.
10. As a user, I want to see a list of the last 10 days with each day's net total, so that I can quickly scan which days had the most activity.
11. As a user, I want to tap a day in the list and see the pie chart update to show that day's category breakdown, so that I can drill into any day's spending.
12. As a user, I want today to be visibly selected by default when the page loads, so that I know which day the pie chart currently represents.
13. As a user, I want the selected day to be visually highlighted in the list (background highlight + left border accent), so that I can easily identify the active day.
14. As a user, I want to tap "Today" in the list to return to today's view after exploring other days, so that I don't get stuck on an old day.
15. As a user, I want the area chart to remain unchanged when I select different days, so that I can maintain context of the overall spending trajectory while drilling into specific days.
16. As a user, I want each section to show its own skeleton loading state, so that I see the page structure taking shape as data loads.
17. As a user, I want partial errors — if one section fails to load, the other sections should still render, so that a single API failure doesn't block the entire page.
18. As a user, I want the pie chart to show "No transactions today" when today has no data yet, so that I'm not confused by a blank page.
19. As a user, I want the area chart to show a flat line at zero when there is no data in the last 10 days, so that the chart doesn't appear broken.
20. As a user, I want the daily list to show all 10 days with zero amounts when there is no data, so that the list structure remains consistent.
21. As a user, I want the pie chart to keep showing its current data while a newly selected day's data is loading, so that I don't experience flicker.
22. As a user, I want the selected day in the list to show a subtle loading indicator while its category data is being fetched, so that I know the app is responding to my tap.

## Implementation Decisions

### API Contract — New Endpoint

**`GET /api/v1/reports/daily?from=&to=`**

Returns daily net amounts (expenses negative, income positive) for the Main book within the given date range.

```json
{
  "data": [
    { "date": "2026-06-11", "amount": -45.00 },
    { "date": "2026-06-12", "amount": 120.00 },
    { "date": "2026-06-13", "amount": -30.00 }
  ]
}
```

- `from` / `to` — ISO 8601 date strings. Both required.
- `date` — `YYYY-MM-DD` string (date only, no time component).
- `amount` — net amount for that day (negative = net expense, positive = net income).
- Sorted ascending by date.
- Days with no transactions are omitted from the response.
- Uses Main book only (same as other reporting endpoints).
- Returns `400` if `from` or `to` is missing.
- Returns `401` if unauthenticated.

### DTO — `DailyReportRow`

```typescript
export interface DailyReportRow {
  date: string   // "YYYY-MM-DD"
  amount: number
}
```

### Modules Built

1. **`InsightDailyPage`** — thin page orchestrator. Uses `useDailyInsight` hook for all state and data fetching. Renders three child components: `InsightComponent` (reused), `DailyAreaChart`, and `DailyList`. No business logic in the component — purely renders hook state into children.

2. **`useDailyInsight` hook** — deep module encapsulating all data fetching, state management, and selection logic for the daily insight page. Exposes a simple interface consumed by the page component. Manages:
   - Fetching today's category breakdown (for the pie's default state)
   - Fetching last N days' daily totals (for the area chart and list)
   - Fetching a selected day's category breakdown on demand
   - Tracking the currently selected day (`null` = today)
   - Splitting category breakdowns into expenses (absolute values) and income (absolute values) for the pie's dual-ring display
   - Loading and error states per data source

   Hook return shape:

   ```typescript
   {
     // Today's pie data
     todayExpenses: Record<string, number>
     todayIncome: Record<string, number>
     isLoadingToday: boolean
     todayError: string | null

     // Daily totals
     dailyTotals: DailyReportRow[]
     accumulatedData: AccumulatedRow[]  // with projection appended
     isLoadingDaily: boolean
     dailyError: string | null

     // Selection
     selectedDay: string | null         // null = today
     selectedDayExpenses: Record<string, number>
     selectedDayIncome: Record<string, number>
     isLoadingSelectedDay: boolean

     // Actions
     selectDay: (date: string | null) => void
   }
   ```

3. **`insightUtils` utilities** — deep module of pure functions for data transformation:
   - `computeAccumulation(rows: DailyReportRow[]): AccumulatedRow[]` — computes running sum from raw daily totals. Expenses (negative amounts) contribute upward to the accumulation (sign flipped); income (positive amounts) pulls the accumulation downward.
   - `computeProjection(accumulated: AccumulatedRow[], M: number): ProjectedRow[]` — computes M-day linear projection based on the average daily change over the historical period. Returns projected points with a `isProjected: true` flag.

   Both functions are pure, stateless, and trivially testable with input/output assertions.

4. **`DailyAreaChart`** — chart component rendering a recharts `AreaChart` with two series:
   - **Historical area**: gradient fill, primary theme color, solid stroke (2px), dots visible on hover only. X-axis shows all 10 dates formatted as `Mon 20` (short weekday + day number).
   - **Trend line**: dotted stroke (`stroke-dasharray`), muted foreground color (`hsl(var(--muted-foreground))`), no area fill. Tooltip labels projected points with "Projected" prefix.
   - Uses the existing shadcn `ChartContainer`, `ChartTooltip`, and `ChartTooltipContent` components.
   - Accepts pre-computed data (accumulated + projected); performs no internal computation. Y-axis auto-scales; chart naturally crosses the Y=0 line when income exceeds expenses.

5. **`DailyList`** — presentational component rendering a vertical list of daily totals. Each row shows:
   - Date label: `"Today"` for the current day, otherwise `"Mon 20"` formatted.
   - Daily net amount, formatted as currency with the existing `formatCurrency` utility.
   - Selected state: `bg-accent` background highlight + a 3px `bg-primary` left border accent.
   - A subtle loading indicator (spinner icon) on the selected row while its category breakdown is being fetched.
   - Receives `dailyTotals`, `selectedDay`, and `onSelectDay` as props.

### Modules Modified

1. **`API.md`** — add the `GET /api/v1/reports/daily` endpoint definition.
2. **`src/types/dto.ts`** — add `DailyReportRow` DTO; re-export from `src/types/index.ts`.
3. **`src/services/reportsService.ts`** — add `getDailyReport(params: { from: string; to: string }): Promise<DailyReportRow[]>`.
4. **`src/mocks/handlers.ts`** — add MSW handler for `GET /api/v1/reports/daily`. Follows the existing pattern: auth check, Main book lookup, filter transactions by date range, group by date (YYYY-MM-DD), sum amounts, return sorted ascending. Omits days with no transactions.
5. **`src/routes/index.tsx`** — add route `/insight/daily` → `InsightDailyPage` as a child of the root `App` layout.

### Constants

- `LAST_N_DAYS = 10` — number of historical days to display. Hardcoded; no UI control.
- `TREND_M_DAYS = 10` — number of projected days for the trend line. Hardcoded; no UI control.
- The frontend computes `from = today - N days` and `to = today` and passes these to both API endpoints.

### Data Flow

1. **Page load**: two parallel fetches — `getCategoryReport({ from: todayStart, to: todayEnd })` for the pie default, and `getDailyReport({ from: N_days_ago, to: todayEnd })` for the area chart and list.
2. **Day tap**: `selectDay(date)` fires `getCategoryReport({ from: dayStart, to: dayEnd })` for that day. The pie shows the previous data until the new fetch resolves (no flicker). The area chart and list are unaffected.
3. **Pie data split**: category rows are split into `expenses` (amount < 0, values made absolute) and `income` (amount > 0, values made absolute). These are passed to `InsightComponent` as `data` and `altData` respectively for the dual-ring display.
4. **Accumulation**: the `computeAccumulation` utility converts raw daily totals into accumulated series data for the area chart.
5. **Projection**: the `computeProjection` utility extends the accumulated series with M projected points, appended to the data passed to `DailyAreaChart`.

### State Handling

- **Loading**: per-section skeletons using the existing shadcn `Skeleton` component — a circular skeleton for the pie, a rectangular skeleton for the area chart, and 10 skeleton rows for the list.
- **Partial errors**: each data source has independent error state. If today's categories fail, the pie shows an error message but the area chart and list render normally. If daily totals fail, the area chart and list show errors but the pie still works.
- **Empty data — pie**: a single gray ring with "No transactions today" rendered in the center.
- **Empty data — area chart**: a flat line at Y=0 with the date axis still displayed; no area fill.
- **Empty data — list**: 10 rows all showing `0.00` with their date labels, preserving the consistent structure.
- **Selected day loading**: the pie keeps showing its current data while the new category breakdown loads. The selected row in the list shows a subtle loading spinner to indicate work in progress.

### Interaction Summary

- The pie chart title is dynamic: "Today" by default, the selected date (e.g., "Mon, Jun 20") when a different day is selected.
- Today is always the first item in the list and is selected by default. Tapping it returns the pie to today's data.
- The area chart never changes when a day is selected — it always shows the full N-day accumulated trajectory.
- Only one day can be selected at a time. Tapping the already-selected day has no effect (except "Today" which re-selects it).

### Routing

```typescript
{
  path: "insight/daily",
  element: <InsightDailyPage />,
}
```

No navigation link is added to any existing page in this PRD. The route is accessible via direct URL only. Navigation wiring between `/insight` and `/insight/daily` is out of scope.

### Existing Component Reuse

The `InsightComponent` (pie chart) is reused as-is. It already supports the `data` (main ring) and `altData` (inner ring) props needed for the expense/income dual-ring display. The daily page passes category amounts with absolute values for both rings.

## Testing Decisions

### Principles

- Test external behavior, not implementation details.
- Use MSW for API mocking (already configured in the project).
- Prefer input/output assertions for pure functions; user-visible assertions for hooks.

### Test Coverage

1. **`insightUtils.test.ts`** — unit tests for pure utility functions:
   - `computeAccumulation`: empty input returns empty array. Single row returns itself with cumulative equal to daily. Multiple rows verify running sum. Negative expenses contribute upward (sign flipped). Positive income pulls accumulation down.
   - `computeProjection`: empty accumulated input returns empty array. Projects M points forward. Average daily rate is correctly derived from the historical data. Projected points carry `isProjected: true`.
   - Prior art: existing utility tests in the codebase follow the same input/output pattern.

2. **`useDailyInsight.test.ts`** — hook test using MSW + React Testing Library's `renderHook`:
   - On mount, fires both initial fetches in parallel.
   - Returns today's expenses/income split correctly from the category report response.
   - Returns daily totals correctly from the daily report response.
   - Returns accumulated data with projection appended.
   - `selectedDay` starts as `null` (today). `selectDay(date)` updates `selectedDay` and triggers a category report fetch for that day.
   - `selectDay(null)` resets to today without a new fetch.
   - Loading states are `true` during fetches and `false` after resolution.
   - Error states are populated when API calls fail; partial successes are preserved.
   - Prior art: existing hook tests in the codebase follow the `renderHook` + MSW pattern.

## Out of Scope

- Navigation between `/insight` (monthly) and `/insight/daily`. The route exists at `/insight/daily` but no tab, link, or button navigates to it.
- Any changes to the existing `InisghtPage.tsx` (monthly insight), including fixing the filename typo.
- Making N (historical days) or M (projection days) configurable via UI or URL parameter. Both are hardcoded constants.
- Any comparison or change-over-time indicators in the pie chart beyond the expense vs. income dual-ring display.
- Stacked area charts by category (the area chart shows only the accumulated net total, not per-category series).
- Persisting the selected day across page navigations.
- Scroll-to or animation when selecting a day in the list.

## Further Notes

- The area chart's accumulation logic inverts expense signs: negative amounts (expenses) contribute upward so the line trends up as spending increases, while positive amounts (income) pull the line back down. Days where income exceeds expenses will cause the accumulated line to cross below Y=0.
- The trend line uses a simple average-daily-rate projection, not linear regression. The average daily change over the N-day historical period is computed and applied linearly to project M days forward.
- All date formatting in the list and area chart X-axis uses the `date-fns` library (already a project dependency) for locale-aware formatting.
- The daily list is always rendered with all 10 days (gaps filled with `amount: 0`), even if the API returns fewer days. This ensures a stable, predictable list height.
- The MSW mock handler for the daily endpoint follows the exact same pattern as the existing handlers for `/api/v1/reports/totals` and `/api/v1/reports/categories` — Main book only, same auth guard, same date filtering logic.
