# 016.dashboard.md — Responsive Dashboard PRD

**Triage:** `ready-for-agent`

---

## Problem Statement

Currently the app's default route (`/`) shows only the Add Transaction page. On wide screens (tablet, desktop), there is significant unused horizontal space. Users who manage finances on larger screens must navigate between separate pages (Add, History, Insights) to get a complete picture of their finances. There is no way to see recent transactions and spending insights alongside the add form without switching tabs.

---

## Solution

Transform the default route `/` into a responsive dashboard that composes up to three pages side by side, adapting to the available viewport width:

| Viewport width | Screens shown |
|---|---|
| `≤ 640px` (mobile) | `AddPage` (current behavior, no wrapper) |
| `> 640px` and `≤ 920px` (tablet) | `[AddPage, HistoryPage]` in `ResponsiveScreens` |
| `> 920px` (desktop) | `[AddPage, HistoryPage, InsightDailyPage]` in `ResponsiveScreens` |

A dedicated `/add` route is introduced so users can always navigate to a standalone Add Transaction page, even from mobile. The Header tab labeled "Add" is not renamed and still points to `/`.

---

## User Stories

1. As a user on a desktop screen, I want to see the Add Transaction form, transaction history, and daily insights side by side on the home page, so that I have a complete financial overview without navigating between tabs.
2. As a user on a tablet, I want to see the Add Transaction form and transaction history side by side on the home page, so that I can enter transactions while reviewing recent activity.
3. As a user on a mobile phone, I want the home page to show only the Add Transaction form (as it does today), so that I have a focused mobile experience.
4. As a user, I want to navigate to `/add` and see only the Add Transaction page, so that I can bookmark it or link to it directly.
5. As a user, I want the Header tab to reflect that `/` is now a Dashboard rather than just "Add", so that the navigation labels match what I see.
6. As a user, I want the standalone routes (`/history`, `/insight/daily`) to remain unchanged, so that I can still navigate to full-screen versions of those pages when I want focused views.
7. As a user on a wide screen, I want the three dashboard panels to share equal width, so that each panel has ample space for its content.
8. As a user resizing my browser window, I want the dashboard to dynamically adjust between single, double, and triple screen layouts, so that the layout always fits my viewport.

## Implementation Decisions

### Module: `pages/dashboard/DashboardPage.tsx`

A new page component at `src/pages/dashboard/DashboardPage.tsx`. Responsibilities:

1. Import and call `useMediaQuery` at two breakpoints:
   - `singleScreen`: `(max-width: 640px)`
   - `doubleScreen`: `(max-width: 920px)`
2. Render based on the widest matching breakpoint:
   - `singleScreen` → `<AddPage />` (plain, no wrapper)
   - `doubleScreen` → `<ResponsiveScreens screens={[<AddPage />, <HistoryPage />]} />`
   - otherwise → `<ResponsiveScreens screens={[<AddPage />, <HistoryPage />, <InsightDailyPage />]} />`

No new store, service, or hook is needed. The existing `useMediaQuery` hook from `@/hooks/use-media-query` and the existing `ResponsiveScreens` component from `@/components/common/responsive/ResponsiveScreens` are composed directly.

### Deep module: `useDashboardBreakpoints` (optional extraction)

If the breakpoint logic grows or needs testing, it can be extracted into a small hook at `src/hooks/useDashboardBreakpoints.ts` returning `{ screenCount: 1 | 2 | 3 }`. For now, the logic is simple enough to live inline in `DashboardPage`.

### Module: Routes — `src/routes/index.tsx`

Changes:

| Before | After |
|---|---|
| Index route `/` → `<AddPage />` | Index route `/` → `<DashboardPage />` |
| — | New route `/add` → `<AddPage />` |

No other routes change.

### Module: Header — `src/components/layout/Header.tsx`

- Rename the "Add" tab label to "Dashboard"
- Keep the tab value as `"/"` (the triggering path)
- Keep the tab value order — it's fine for "Dashboard" to remain in the same position as "Add" was

### InsightDailyPage back button

The existing back button in `InsightDailyPage` that navigates to `/insight` remains visible even when rendered inside the dashboard. Clicking it navigates the user away from the dashboard to the standalone insight page — this is acceptable behavior.

### Data sharing

All three pages (`AddPage`, `HistoryPage`, `InsightDailyPage`) already use shared Zustand stores (`useTransactionsStore`, `useBooksStore`, `useCategoriesStore`). When rendered side by side in the dashboard:
- Creating a transaction via `AddPage` automatically updates the list shown in `HistoryPage`
- No additional synchronization logic is needed
- Each page instance manages its own local state independently (e.g., `useDailyInsight` hook state for `InsightDailyPage`)

### ResponsiveScreens and the `.multi-screen` CSS class

`ResponsiveScreens` already calls `useMultiScreen(true)` when more than one screen is shown. The existing `.multi-screen` class is applied to `<html>` and handled by any existing CSS. No CSS changes are expected.

## Testing Decisions

### Testing philosophy

Tests should verify external behavior, not implementation details. Focus on:
- Which screens are rendered at each breakpoint
- That the `/add` route renders `AddPage` independently
- That the Header tab label is correct
- That the dashboard renders without errors when wrapped in appropriate mock viewport sizes

### Module: DashboardPage

- **Unit test `DashboardPage.test.tsx`**: Mock `useMediaQuery` to return different values for each breakpoint. Assert that:
  - When `singleScreen` (≤640px): only `AddPage` is rendered, no `ResponsiveScreens`
  - When `doubleScreen` (>640px, ≤920px): `ResponsiveScreens` renders with 2 children
  - When `wide` (>920px): `ResponsiveScreens` renders with 3 children
- Use `screen.getByTestId` or text matchers to verify the presence of each page's distinctive content.

### Module: Routes

- **Router integration test**: Verify that navigating to `/` renders dashboard content, and navigating to `/add` renders AddPage content without surrounding dashboard elements.

### Module: Header

- **Component test**: Verify the tab label displays "Dashboard" not "Add".

### Prior art in the codebase

Follow patterns established by:
- `InsightDailyPage.test.tsx` — page-level rendering tests
- `EditTransactionPage.test.tsx` — route/page interaction testing
- `HistoryPage.test.tsx` — component rendering with store mocking

### Mocking strategy

- `window.matchMedia` must be mocked for tests that exercise `useMediaQuery`. The existing test setup at `src/test-setup.ts` already includes a `matchMedia` mock — this may need extension to support custom query strings and return values.
- Zustand stores should use their real implementations with mocked underlying services, or use `vi.fn()` mocks for individual store methods.

---

## Out of Scope

- New breakpoints beyond the three defined (640px, 920px)
- Resizable / draggable column dividers between panels
- Customization of which pages appear in the dashboard
- Remembering the user's preferred layout
- Animated transitions when switching between screen counts
- Syncing scroll positions between panels
- Adding new Header tabs or restructuring the Header navigation
- Changing the behavior of standalone pages (`/history`, `/insight`, `/insight/daily`, etc.)
