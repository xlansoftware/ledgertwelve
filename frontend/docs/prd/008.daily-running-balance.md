# PRD 008 — Daily Running Balance

## Problem Statement

The Daily Insight page (`/insight/daily`) currently shows an accumulated spending curve that starts at **zero** on the 1st of the current month and builds day by day. This means the Y-axis values represent "how much you've spent since the month started" — not your actual account balance.

As the year progresses, this becomes increasingly misleading. In June, a user who has spent $30,000 YTD sees a chart showing only ~$2,000 accumulated. The curve resets to zero each month, losing all context of the year's financial trajectory. The forward projection (end-of-month estimate) is also wrong because it's based on only the current month's data, not the full year's run rate.

Users need the accumulated chart to reflect their **true year-to-date balance** — starting from their balance at the beginning of the year and passing through every day up to today.

## Solution

Introduce an `?asOf=` parameter to the existing `GET /api/v1/books/{bookId}/stats` endpoint so the frontend can fetch the net balance as of any arbitrary date.

The `useDailyInsight` hook will:

1. Fetch the balance as of **the last day of the previous month** using the stats endpoint with `?asOf=`.
2. Fetch daily totals from **the 1st of the current month** through **today** (existing behavior, unchanged range).
3. Use the fetched balance as the **seed value** for the accumulation, so the accumulated curve represents the true running balance.

The accumulation sign convention changes: instead of flipping expense signs so that spending pushes the curve up, the curve now tracks the actual account balance. Expenses (negative amounts) reduce the balance; income (positive amounts) increases it.

The chart label changes from **"Accumulated Spending"** to **"Running Balance"** to reflect the new semantics.

## User Stories

1. As a user, I want the accumulated chart on the daily insight page to show my actual account balance day by day, so that I can see my true financial position rather than just month-to-date spending.
2. As a user, I want the balance to start from my actual position at the beginning of the year, so that the chart reflects my full annual financial trajectory.
3. As a user, I want expenses to correctly decrease the balance and income to increase it, so that the curve accurately represents what I have left (or owe).
4. As a user, I want the forward projection to be based on my full year-to-date spending rate, so that the end-of-month estimate is more accurate.
5. As a user, I want the chart title to clearly say "Running Balance" instead of "Accumulated Spending," so that I know what the curve represents.
6. As a user, I want the tooltip to label the accumulated value as "Balance" instead of "Total," so that the terminology is consistent.
7. As a user, I don't want to see any additional loading delay while the opening balance is fetched — it should happen in parallel with existing fetches.
8. As a user, I want partial errors: if the balance fetch fails, the chart should still show data starting from zero (graceful degradation), so that a single API failure doesn't break the entire page.
9. As a developer, I want the stats service interface to accept an optional `asOf` parameter, so that any consumer can query historical balances.

## Implementation Decisions

### API Contract — Modified Endpoint

**`GET /api/v1/books/{bookId}/stats`**

Existing endpoint. New optional query parameter:

```text
asOf=2026-05-31   (optional ISO 8601 date string)
```

- If `asOf` is **omitted**, behavior is unchanged: returns stats for the entire book (all transactions).
- If `asOf` is **provided**, `transactionCount` and `totalSum` are computed from transactions **on or before** that date only.
- The `totalSum` is the net sum of all non-closing transactions up to and including that date.

Response shape (unchanged):

```json
{
  "data": {
    "transactionCount": 142,
    "totalSum": 15234.50
  }
}
```

Errors (unchanged):

```json
// 404 — Book not found or not visible
{ "error": "Book not found" }
```

### Modules Modified

#### 1. `API.md` — Update stats endpoint docs

Add the `asOf` query parameter to the `GET /api/v1/books/{bookId}/stats` section.

#### 2. `src/types/dto.ts` — No change needed

`BookStatsDto` already has `{ transactionCount: number; totalSum: number }`. The shape does not change.

#### 3. `src/services/booksService.ts` — Add optional `asOf` parameter

```typescript
export interface GetBookStatsParams {
  asOf?: string   // "YYYY-MM-DD"
}

export async function getBookStats(
  bookId: string,
  params?: GetBookStatsParams,
): Promise<BookStatsDto>
```

The `asOf` value is passed as a query parameter. When omitted, behavior is unchanged.

#### 4. `src/mocks/handlers.ts` — Update stats handler

The existing `GET /api/v1/books/{bookId}/stats` handler will be updated to:

- Parse the optional `asOf` query parameter.
- If present, filter the Main book transactions to include only those with `dateTime <= asOf`.
- Compute `transactionCount` and `totalSum` from the filtered set.
- If absent, retain existing behavior (all transactions).

#### 5. `src/pages/insight/insightUtils.ts` — Change sign convention and add seed

```typescript
export function computeAccumulation(
  rows: { date: string; amount: number }[],
  initialCumulative?: number,  // new, default 0
): AccumulatedRow[]
```

Changes:
- Sign convention: `cumulative += row.amount` (was `cumulative += -row.amount`). Expenses (negative) correctly decrease the balance, income (positive) increases it.
- Optional `initialCumulative` parameter seeds the starting cumulative value.
- When omitted or `undefined`, defaults to `0` (backward-compatible for existing callers).

`computeProjection` requires no changes — it works with any cumulative series regardless of scale or sign.

#### 6. `src/pages/insight/useDailyInsight.ts` — Add stats fetch and seed

New state:

```typescript
const [openingBalance, setOpeningBalance] = useState<number | null>(null)
const [isLoadingBalance, setIsLoadingBalance] = useState(true)
const [balanceError, setBalanceError] = useState<string | null>(null)
```

New `useEffect`:

```typescript
// Fetch opening balance as of last day of previous month
useEffect(() => {
  const asOf = lastDayOfPreviousMonth(todayDate)  // "YYYY-MM-DD"
  setIsLoadingBalance(true)
  setBalanceError(null)

  getBookStats("book_main", { asOf })
    .then((stats) => {
      setOpeningBalance(stats.totalSum)
      setIsLoadingBalance(false)
    })
    .catch((err) => {
      setBalanceError(err instanceof Error ? err.message : "Failed to load opening balance")
      setOpeningBalance(null)
      setIsLoadingBalance(false)
    })
}, [todayDate])
```

Modified accumulation logic in the existing `useMemo`:

```typescript
const accumulatedData = useMemo<(AccumulatedRow | ProjectedRow)[]>(() => {
  const seed = openingBalance ?? 0  // graceful fallback
  const accumulated = computeAccumulation(filledDailyTotals, seed)
  const projection = computeProjection(accumulated, trendMDays)
  return [...accumulated, ...projection]
}, [filledDailyTotals, trendMDays, openingBalance])
```

Key behaviors:
- The stats fetch runs in **parallel** with the existing today/daily fetches.
- If the stats fetch fails, `openingBalance` stays `null`, and the accumulation falls back to `seed = 0` (graceful degradation; balance starts from zero as currently).
- The opening balance is a single number (not an array), so it adds negligible overhead.

Helper function:

```typescript
function lastDayOfPreviousMonth(date: Date): string {
  // Returns "YYYY-MM-DD" for the last day of the month before `date`'s month
  const d = new Date(date.getFullYear(), date.getMonth(), 0)  // day 0 = last day of previous month
  return d.toISOString().slice(0, 10)
}
```

#### 7. `src/pages/insight/DailyAreaChart.tsx` — Update title

Change the chart heading from `"Accumulated Spending"` to `"Running Balance"`.

Change the tooltip label from `"Total: {formatCurrency(...)}"` to `"Balance: {formatCurrency(...)}"`.

### Data Flow

```
Page mount (parallel fetches):
  ├── getBookStats("book_main", { asOf: "2026-05-31" })
  │     → openingBalance = 15234.50
  ├── getCategoryReport({ from: todayStart, to: todayEnd })
  │     → todayExpenses / todayIncome
  └── getDailyReport({ from: "2026-06-01", to: "2026-06-20" })
        → dailyTotals

Accumulation:
  computeAccumulation(dailyTotals, openingBalance)
    → cumulative starts at 15234.50
    → June 1:  15234.50 + (-45)   = 15189.50
    → June 2:  15189.50 + (-30)   = 15159.50
    → June 3:  15159.50 + 200     = 15359.50
    → ...up to today

Projection:
  computeProjection(accumulated, trendMDays)
    → based on average daily change from June 1 → June 20
    → projected values extend from today's balance forward
```

### Existing Constants

- `book_main` — hardcoded book ID (consistent with reporting endpoints).
- `trendMDays = daysInMonth - lastNDays` — unchanged; projection covers the remainder of the current month.

### Graceful Degradation

If the `getBookStats` call fails (network error, 404, 500):

| Scenario | Effect |
|----------|--------|
| Stats fetch fails | `openingBalance = null`, seed = 0, chart starts at zero. The curve still tracks day-by-day changes correctly, just without the YTD offset. |
| Daily fetch fails | Chart shows error state (unchanged behavior). |
| Stats + daily both fail | Chart shows error state (unchanged behavior). |

The daily totals fetch is unaffected by the stats fetch — they are independent async operations.

### Interaction Summary (Unchanged)

- The pie chart, daily list, and day selection behavior are unchanged.
- The accumulated chart is the only visual that changes.
- Loading skeleton for the chart still covers the daily fetch; the stats fetch is fast (single number) and completes within the same lifecycle.
- No new UI elements (no "as of" label, no balance indicator outside the chart).

## Out of Scope

- Changing the pie chart, daily list, or day selection behavior.
- Making the book ID configurable (e.g., book selector). The Main book is hardcoded.
- Exposing the opening balance value in the UI outside the chart (no "Balance as of May 31" label).
- Making `asOf` configurable via URL parameter or UI control.
- Changing the projection algorithm or period. The projection still covers the remainder of the current month.
- Any performance optimization (the stats fetch is a single number, negligible overhead).
- Adding a loading skeleton specifically for the opening balance — it loads in parallel with existing fetches.

## Further Notes

- The `lastDayOfPreviousMonth` helper uses JavaScript's Date day-0 trick: `new Date(year, month, 0)` returns the last day of the previous month. This is a well-established pattern and avoids date arithmetic bugs for month boundaries.
- The sign convention change (`cumulative += row.amount`) is the correct financial semantics: starting balance + net flow = ending balance. This also makes the chart intuitive — the line goes down when you spend, up when you earn.
- The `?asOf=` query parameter filtering on the backend should use the inclusive date boundary (`<=`), meaning transactions occurring exactly on that date are included. This ensures the balance is "as of end of day" on the specified date.
- The MSW handler should mirror the backend logic exactly: filter non-closing Main book transactions by `dateTime <= asOfDate`, then sum amounts.
