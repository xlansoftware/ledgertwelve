# 015.import.md — Import Feature PRD

**Triage:** `ready-for-agent`

---

## Problem Statement

Users need to bring financial data from external sources (bank exports, other accounting tools, spreadsheet exports) into Ledger Twelve. Without an import feature, every transaction, category, and book must be entered manually through the UI — which is impractical for migrating from another tool, restoring a backup, or batch-importing bank statements. Users also need a way to restore data from the backup files generated by the existing export feature.

---

## Solution

A multi-step import workflow accessible from Settings and the Export page:

1. **Choose what to import** — transactions, categories, books, or a full backup restore.
2. **Upload a file** — CSV, XLSX, or JSON (or a backup JSON for restore).
3. **Map columns** — the app auto-discovers source column names, suggests target field mappings using heuristics, and lets the user correct them via dropdowns. Unmapped columns are ignored. Preview values from the first rows show how data will be parsed.
4. **Preview results** — the server validates all rows without committing. The user sees counts of what will be created, updated, or skipped (with per-row error details), and can go back to adjust the mapping.
5. **Confirm import** — partial-success: valid rows are committed, failed rows are skipped and reported.

The feature works both **online** (via `POST /api/v1/imports`) and **offline** (via IndexedDB), following the existing `ServiceFactory` pattern.

---

## User Stories

1. As a user migrating from another finance app, I want to import my transactions from a CSV file, so that I don't have to re-enter hundreds of entries manually.
2. As a user, I want the app to automatically detect which column is the date, which is the amount, and which is the description, so that I don't have to map every column by hand.
3. As a user, I want to review and correct the automatic column mapping before importing, so that I can fix any misdetected columns.
4. As a user, I want to see a preview of how my data will look after import, so that I can verify the parsing is correct before committing.
5. As a user, I want to import categories from a file, so that I can set up my category list in bulk instead of creating them one by one.
6. As a user, I want to import books from a file, so that I can create multiple ledger books at once.
7. As a user, I want to restore a full backup (JSON) created by the export feature, so that I can recover my data after switching devices or browsers.
8. As a user restoring a backup, I want to see a summary of what will be created vs updated before the restore happens, so that I understand the impact.
9. As a user, I want the import to be partial-success — valid rows are imported and invalid rows are reported with specific error messages, so that I can fix problems in my source file without losing the good data.
10. As a user, I want to see exactly which rows failed and why, so that I can fix them in my source file and re-import.
11. As a user, I want to optionally clear all existing data of a type before importing, so that I can do a clean bulk replacement (e.g., replace all categories with a new set).
12. As a user with a multi-sheet XLSX file, I want to choose which sheet to import from, so that I can import the right data.
13. As a user, I want to be able to go back from the preview page to the mapping page if something looks wrong, so that I can adjust my mapping without starting over.
14. As a user, I want to be able to go back from the mapping page to the upload page if I picked the wrong file or entity type, so that I don't have to restart the entire import.
15. As a user, I want the import to work in offline mode (using my local IndexedDB), so that I can import data without an internet connection.
16. As a user, I want a confirmation dialog before clearing existing data, so that I don't accidentally delete my data.
17. As a user, I want the import feature to be discoverable from the Settings page and the Export page, so that I can find it when I need it.
18. As a user importing a JSON file of transactions, I want the same column mapping flow as CSV, so that I can import structured data from other tools that export JSON.
19. As a user, I want the import to support upsert (update if the record already exists by ID, create otherwise), so that I can re-import corrected data without creating duplicates.
20. As a user, I want to be warned about unsupported backup versions, so that I know a backup was created by a newer version of the app and may not import correctly.

## Implementation Decisions

### Architecture: Pages and routing

Three dedicated routes under `/import`, each a separate page navigated sequentially with back buttons:

- `/import` — entity type selection, file upload, sheet selector (for multi-sheet XLSX), "clear existing" checkbox
- `/import/mapping` — column mapping table (card layout on mobile), with auto-mapping pre-fills
- `/import/preview` — server preview results, issue list, confirm import button, completion state

Backup restore skips `/import/mapping` — the flow is `/import` → `/import/preview`.

### Architecture: Navigation and entry points

- Import section added to `SettingsPage` (button navigates to `/import`)
- Reciprocal link from Export page
- No new tab in the main header navigation

### Architecture: State management

A new Zustand store (`useImportStore`) holds cross-page state:

- `step`: `'upload' | 'mapping' | 'preview' | 'importing' | 'complete'`
- `entityType`: `'transactions' | 'categories' | 'books' | 'backup' | null`
- `bookId`: fallback target book for transactions
- `clearExisting`: boolean
- `fileName`: original file name
- `parsedColumns`, `parsedRows`: raw data (source column keys, string values)
- `mapping`: source column → target field (null = ignored)
- `convertedRows`: target field keys, typed values
- `sheetNames`, `selectedSheet`: for multi-sheet XLSX
- `previewResult`: server response (counts + issues)
- `backupData`: parsed backup JSON (backup restore only)

State is preserved during back navigation within the import flow. Navigating away from import entirely preserves state; re-entering `/import` prompts to resume or start over. Refreshing on mapping/preview redirects to `/import` (state lost).

### Architecture: Service factory pattern

Import follows the existing `ServiceFactory` pattern:

- `IImportService` interface with `importData(req: ImportRequest): Promise<ImportResult>`
- `OnlineImportService` — delegates to `POST /api/v1/imports`
- `OfflineImportService` — validates against IndexedDB data, then writes via existing offline service classes
- Added to `ServiceFactory` and both `createOnlineFactory()` / `createOfflineFactory()`

### Architecture: Offline import execution

The `OfflineImportService` replicates server-side validation and writes locally:

1. Validates all rows against IndexedDB data (existing IDs, category/book references, required fields)
2. If `clearExisting: true`, deletes existing records from IndexedDB (with Main book protection)
3. Iterates rows: creates or updates via `OfflineTransactionsService`/`OfflineCategoriesService`/`OfflineBooksService`
4. Processing order for backup restore: books → categories → transactions
5. Returns `ImportResult` with counts and per-row issues

### Parsing: Client-side file parsing

All parsing happens client-side before any server interaction:

- **CSV**: PapaParse library. Handles quoted fields, embedded delimiters, line breaks. UTF-8 assumed.
- **XLSX**: ExcelJS library. Sheet discovery with user-selectable sheet picker when multiple sheets exist.
- **JSON**: `JSON.parse()`. Accepted for all entity types as an array of flat objects (keys are column names). Backup restore expects the specific backup schema.

Both libraries are lazy-loaded (import page is not a hot path).

### Parsing: Date conversion

Best-effort date parsing for columns mapped to `dateTime`:

1. ISO 8601 (`2026-06-24T12:00:00Z`, `2026-06-24`)
2. Common separator formats (`06/24/2026`, `24.06.2026`, `24-Jun-2026`)
3. Excel serial numbers (integers ~30000–100000, with leap-year bug compensation)
4. `new Date()` as last resort

Unparseable dates are flagged as warnings. The mapping page shows parsed preview values so the user can verify.

### Parsing: Number conversion

Best-effort number parsing for columns mapped to `amount`, `originalAmount`, `exchangeRate`:

- Strip currency symbols (`$`, `€`, `£`)
- Handle parenthetical negatives: `(100.50)` → `-100.50`
- Detect comma vs dot decimal separator by prevalence in the column
- Handle thousands separators (comma, space, dot)
- Handle Excel accounting `-` for zero
- Empty strings → `null`/`undefined` (warned)
- Unparseable → error

### Column mapping: UI

Table layout on desktop:

| Source Column | Preview (first 3 values) | Maps To |
|---|---|---|
| Date | 2026-01-01, 2026-01-02, 2026-01-03 | [dateTime ▾] |
| Value | -100, -50, -25 | [amount ▾] |

Card layout on mobile (≤768px): each source column becomes a stacked card with label, preview values, and dropdown.

- Empty dropdown = column is ignored
- Row count shown ("142 rows found")
- Required fields highlighted
- For transactions: book picker dropdown above the mapping table (fallback book; overridden if `bookId` column is mapped)

### Column mapping: Auto-mapping heuristics

Case-insensitive exact match + curated alias table. Each target field has priority-ordered aliases. If a source column matches multiple targets' aliases, the first match wins.

Alias table (not exhaustive — documented as a decision point):

- `dateTime`: `["date", "datetime", "date time", "transaction date", "posting date", "date_posted", "value date"]`
- `amount`: `["amount", "value", "sum", "price", "total", "debit", "credit", "transaction amount"]`
- `note`: `["note", "description", "memo", "details", "comment", "remarks", "narration"]`
- `categoryName`: `["category", "category name", "type", "group", "classification"]`
- `id`: `["id", "transaction id", "tx id", "reference", "ref"]`
- And others for `originalCurrency`, `originalAmount`, `exchangeRate`, `bookId`, `name`, `recurring`, `color`, `icon`, `currency`, `status`

### Import semantics: Upsert vs create

- If `id` column is mapped → per-row behavior: rows with non-empty `id` matching an existing record are updated; rows with empty/null `id` or unmatched `id` are created
- If `id` column is not mapped → all rows are created
- Applies to all entity types

### Import semantics: clearExisting

- Checkbox on the upload page (not mapping page), with confirmation dialog at preview time
- For transactions: deletes all transactions in the target book before importing
- For categories: deletes all user categories
- For books: deletes all user books except Main (silently preserved)
- For backup restore: clears transactions and categories only; books are never cleared (always merged by ID)
- In preview mode: `deleted` count reflects what would be deleted, no actual deletion

### Import semantics: Closed books

Importing into a closed book is allowed. No restriction.

### Import semantics: Partial success

Import is never all-or-nothing. Rows that pass validation are committed; failed rows are skipped. The response includes counts and per-row error/warning details. The user can fix errors in the source file and re-import (upsert via ID handles already-imported rows).

### API: POST /api/v1/imports

Single endpoint serving all import modes. Full specification in `API.md`. Key design points:

- `preview: true` validates without committing; `preview: false` commits
- `entityType` determines which validation rules apply
- `mapping` is metadata-only (for error messages); client applies mapping before sending
- `rows` use target field names with already-typed values
- Response includes `created`, `updated`, `deleted` (when clearExisting), `errors`, `warnings`, and `issues[]`
- Row indices in issues are 1-based
- Backup restore response nests counts per entity type (`books`, `categories`, `transactions`)

### API: Version handling for backup restore

- Version-specific import/preview methods dispatched by backup `version` field
- Version `1` is the currently supported version
- Unsupported versions: import blocked with error message warning the user

### MSW: Mock handler

Light simulation handler for `POST /api/v1/imports`:

- Returns plausible counts based on request size
- Hardcoded "bad" values trigger error rows (e.g., `amount: "error"` → error issue)
- Does not modify mock state
- Tests can override with `server.use()` for specific scenarios

### Edge cases

- **Empty file / no data rows**: Error on upload page, don't advance to mapping
- **All columns ignored**: Import button disabled with message "At least one column must be mapped"
- **CSV/XLSX/JSON parse failure**: Clear error message on upload page
- **JSON is array of non-objects**: Error (no columns to map)
- **Backup JSON missing required fields**: Schema validation error
- **File with no sheet data (XLSX)**: Similar to empty file
- **Concurrent imports**: Blocked — only one import at a time (`step === 'importing'` prevents new import)
- **Network failure during online import**: Retry button on preview/completion page
- **Main book protection**: Never deleted, even with `clearExisting: true`

### File constraints

- No file size limit enforced
- No row count limit enforced
- No row exclusion or reordering in the UI

### Loading states

- File parsing: simple spinner with "Analyzing file..."
- Import execution: button shows loading state

### UI: Completion state

After successful import, the preview page transitions to completion state showing:
- Final counts (created, updated, deleted, errors)
- Any skipped rows with error details
- Navigation links: "Go to History", "Import more", "Back to Settings"

## Testing Decisions

### Testing philosophy

Tests should verify external behavior, not implementation details. Focus on:
- What the user sees and can do
- Correct data transformations
- Error states are surfaced appropriately
- The flow between pages preserves and resets state correctly

### Deep modules to test in isolation

These pure-function modules are ideal candidates for isolated unit tests:

- **Parser module** (`parser.ts`): Takes a `File` (or string/blob for CSV) and returns `{ columns: string[], rows: Record<string, string>[] }`. Test with sample CSV strings, XLSX buffers, and JSON strings. Verifies correct column extraction, row parsing, multi-sheet discovery.
- **Mapper module** (`mapper.ts`): Takes `{ columns: string[], entityType }` and returns `{ mapping: Record<string, string | null> }`. Tests verify alias matching, case-insensitivity, priority ordering, and no-match (null) cases.
- **Converter module** (`converter.ts`): Takes `{ rawRows, mapping, entityType }` and returns `{ convertedRows: Record<string, unknown>[] }`. Tests verify date parsing (ISO, slash, dot, Excel serial), number parsing (negatives, parenthetical, currency symbols, European format), and handling of null/empty values.

### Integration tests

- **`useImportStore`**: Tests the state machine (step transitions, data preservation on back navigation, reset behavior)
- **`importsService`**: Tests the HTTP layer — correct request construction and response parsing

### Component tests

- **`ImportPage`**: File upload acceptance, entity type selection, sheet picker display, clearExisting checkbox, navigation to mapping
- **`MappingPage`**: Auto-mapping display, dropdown interaction, back navigation, mobile layout switch
- **`PreviewPage`**: Preview result display, issue list rendering, import button behavior, completion state, clearExisting confirmation dialog

### Prior art in the codebase

Follow the patterns established by:
- `exportsService.test.ts` — service-level tests with MSW
- `useBooksStore.test.ts` — Zustand store testing
- `useDailyInsight.test.ts` — hook orchestration testing
- `ExportPage` structure — page composition with hook separation

## Out of Scope

- Pasting tabular data from clipboard
- Direct bank API connections (Plaid, etc.)
- Importing from other accounting software formats (QuickBooks, Xero)
- Merging duplicate transactions beyond ID-based upsert
- Editing individual cell values within the mapping UI
- Saving/loading mapping templates for reuse
- Scheduled/automated imports
- Importing reports or historical exchange rates
- CSV encoding auto-detection beyond UTF-8
- Row-level deduplication (only ID-based upsert)
- Version migration for old backup formats (only version 1 supported)
