# PRD 010 — Long-window Average for Insight Projections

## Problem Statement

The Daily and Monthly insight pages compute their projection lines using a short historical window:

- **Daily**: Average daily net change is computed from the current month so far (1–31 days). If the user is on June 5, only 5 days of data inform the projection for the remaining 25 days.
- **Monthly**: Average monthly net change is computed from the current year so far (1–11 months). If the user is in June, only 6 months of data inform the projection for the remaining 6 months.

This produces volatile projections early in the period. A single outlier day or month disproportionately skews the projected year‑end or month‑end balance. Users get a misleading picture of their financial trajectory.

## Solution

Introduce two new read‑only API endpoints that return a **net average** computed over a wide rolling window:

- `GET /api/v1/reports/average/daily` — returns the average daily net amount over the last 365 days.
- `GET /api/v1/reports/average/monthly` — returns the average monthly net amount over the last 12 months.

The hooks (`useDailyInsight`, `useMonthlyInsight`) fetch the wide‑window average in parallel with existing data fetches. The projection is then computed from this externally provided average rather than from the short‑window historical data. If the average API call fails, the hook falls back to the current short‑window behaviour — graceful degradation.

The historical display area chart, pie chart, and period list remain unchanged. Only the **projection line** becomes statistically more meaningful.

## User Stories

1. As a user viewing the Daily Insight page, I want the projection line to be based on my average daily net over the last 365 days, so that the forecast is not skewed by a few unusual days at the start of the month.
2. As a user viewing the Monthly Insight page, I want the projection line to be based on my average monthly net over the last 12 months, so that the forecast is not skewed by a few unusual months early in the year.
3. As a user, I want the page to still work even if the average API is unavailable, so that I never see a broken page — the projection falls back to the current short‑window calculation.
4. As a developer, I want a well‑defined contract for the average endpoints, so that the backend can implement them independently.
5. As a maintainer, I want the projection logic to be a pure function that can be tested in isolation, so that the averaging change is easy to verify and does not regress.

## Implementation Decisions

### API Contract — Two New Endpoints

#### `GET /api/v1/reports/average/daily`

**Purpose**: Returns the average daily net amount over a date range. Used to compute the projection rate for the daily insight chart.

**Query Parameters**:

| Parameter | Required | Format | Description |
|-----------|----------|--------|-------------|
| `from` | yes | `YYYY-MM-DD` | Inclusive start date. |
| `to` | yes | `YYYY-MM-DD` | Exclusive end date. |

**Semantics**: Server computes `SUM(amount) / COUNT(days_with_transactions)` for Main‑book transactions whose `date` is `>= from` and `< to`. Days with no transactions are not counted — they contribute zero to the sum but do not inflate the divisor.

**Response**:

```json
{
  "data": {
    "average": -45.50,
    "count": 365
  }
}
```

- `average` — net average per day (negative = net expense, positive = net income). Rounded to 2 decimal places.
- `count` — number of days in the queried range that have at least one transaction.

**Errors**:

```json
// 400 — Missing from or to
{ "error": "from and to query parameters are required" }

// 401 — Unauthenticated
{ "error": "Unauthorized" }
```

**Example**:

```text
GET /api/v1/reports/average/daily?from=2025-06-22&to=2026-06-22
```

---

#### `GET /api/v1/reports/average/monthly`

**Purpose**: Returns the average monthly net amount over a date range. Used to compute the projection rate for the monthly insight chart.

**Query Parameters**:

| Parameter | Required | Format | Description |
|-----------|----------|--------|-------------|
| `from` | yes | `YYYY-MM-DD` | Inclusive start date. |
| `to` | yes | `YYYY-MM-DD` | Exclusive end date. |

**Semantics**: Server groups transactions by calendar month (`YYYY-MM`) within `[from, to)`, computes the net sum for each month, then returns `SUM(monthly_net) / COUNT(months_with_transactions)`. Months with no transactions are not counted.

**Response**:

```json
{
  "data": {
    "average": -1380.00,
    "count": 12
  }
}
```

- `average` — net average per month (negative = net expense, positive = net income). Rounded to 2 decimal places.
- `count` — number of months in the queried range that have at least one transaction.

**Errors**:

```json
// 400 — Missing from or to
{ "error": "from and to query parameters are required" }

// 401 — Unauthenticated
{ "error": "Unauthorized" }
```

**Example**:

```text
GET /api/v1/reports/average/monthly?from=2025-06-01&to=2026-06-01
```

### Date range convention

Both endpoints follow the project-wide convention (from API.md):

- `from` is **inclusive** (`>=`)
- `to` is **exclusive** (`<`)

The frontend computes:

| Endpoint | `from` | `to` | Coverage |
|----------|--------|------|----------|
| daily | `today - 365 days` | `today` | 365 rolling days ending yesterday |
| monthly | `12 months ago (1st)` | `1st of current month` | 12 rolling months ending last month |

### Utility — pure projection function

A new function `computeProjectionFromAverage` is added to `insightUtils.ts`. It is a **deep module**: it encapsulates the projection logic behind a simple, testable interface that takes three numeric inputs and returns an ordered list of projected points.

```typescript
export function computeProjectionFromAverage(
  lastCumulative: number,
  avgChange: number,
  M: number,
): ProjectedRow[]
```

- `lastCumulative` — the cumulative balance at the last historical data point.
- `avgChange` — the net average change per period (from the API).
- `M` — the number of projected periods forward.

Returns `ProjectedRow[]` with dates starting from the day/month after the last historical point, daily=`avgChange`, cumulative extending from `lastCumulative` by `avgChange` each step.

The existing `computeProjection` function remains unchanged for the fallback path.

### Hook orchestration

**`useDailyInsight.ts`** and **`useMonthlyInsight.ts`** both gain:

| New state | Type | Description |
|-----------|------|-------------|
| `averageChange` | `number \| null` | The net average from the API |
| `isLoadingAverage` | `boolean` | Loading state for the average fetch |
| `averageError` | `string \| null` | Error state for the average fetch |

**Fetching**: The average API is called in parallel with the existing fetches (daily report / monthly report, book stats, category report). No sequential dependencies.

**Projection logic (inside the `accumulatedData` `useMemo`)**:

```
if averageChange !== null:
    use computeProjectionFromAverage(lastCumulative, averageChange, M)
else:
    use computeProjection(accumulated, M)  // fallback to current behaviour
```

The `lastCumulative` is the cumulative value of the last accumulated historical row. `M` is the remaining periods (days in month / months in year), unchanged from current logic.

**Graceful degradation**: If the average API call fails (network error, server error, any exception), `averageChange` stays `null`, and the `useMemo` falls through to the existing `computeProjection` using the short‑window average. No UI error state is shown for the average fetch — the user sees the same chart as today.

## Testing Decisions

### What makes a good test

- Tests focus on **external behaviour**, not implementation details.
- Pure functions are tested by input/output pairs — no mocking needed.
- Hook tests use MSW-mocked API responses and assert on returned state and derived data.
- Error paths are tested (API failure → fallback behaviour).
- Edge cases: empty data, single data point, zero average, negative periods (`M=0`).

### Modules tested

| Module | Test file | Type | Notes |
|--------|-----------|------|-------|
| `computeProjectionFromAverage` | `insightUtils.test.ts` | Unit (pure function) | Input/output table. No mocks. |
| `useDailyInsight` | `useDailyInsight.test.ts` | Hook (React Testing Library) | Mock `getDailyAverage`, `getDailyReport`, `getCategoryReport`. Verify `accumulatedData` uses external average. |
| `useMonthlyInsight` | `useMonthlyInsight.test.ts` | Hook (React Testing Library) | Same pattern as daily tests. |
| Error fallback in both hooks | same test files | Scenario tests | Mock `getDailyAverage`/`getMonthlyAverage` to reject. Verify projection still renders using short‑window average. |

### Prior art

The existing tests for `computeProjection` in `insightUtils.test.ts` follow the same pure-function pattern (input → expected output). The hook tests in `useDailyInsight.test.ts` and `useMonthlyInsight.test.ts` already mock service functions and test loading states, error handling, and data returns — the new tests follow the same conventions.

## Out of Scope

- **UI changes** — No new components, no chart modifications, no stat cards displaying the average number. The average is an invisible improvement to the projection calculation.
- **Parameters beyond `from`/`to`** — No `bookId`, no `category`, no `period` granularity filter. Main book only, net only.
- **Caching or deduplication** — The average is fetched on every mount / route visit. No client-side caching strategy is introduced in this PRD.
- **Backend implementation** — This PRD defines the API contract only. Backend implementation is separate work that must match the contract.
- **Changing the historical display window** — The area chart and list continue to show the current month (daily) / year-to-date (monthly). Only the projection rate changes.
