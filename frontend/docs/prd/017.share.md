# PRD — Global Book Sharing

## Problem Statement

Users of ledger12 can create multiple books (e.g., "Main", "Vacation 2026") to organize their finances. Currently, there is no way to share a book with another person. When a user wants to collaborate on expenses with a family member, partner, or roommate, they have no mechanism to give that person access to their books. The only option today is to export data and share it externally — a manual, one-time snapshot that doesn't enable real-time collaboration.

The user needs a way to invite others to view and edit their books from within the application.

## Solution

Introduce a **global sharing** model: when a user shares with another registered user, that person gets `edit` access to **every book the sharer owns**, both existing books and any books created in the future. This avoids the complexity of per-book permission management at this stage. Later, per-book sharing with granular permissions (`view` / `edit`) will be added as a follow-up feature.

The feature adds:

- A **Shared Users** section on the Settings page that shows how many users have access, with a link to the dedicated Share page.
- A **Share page** (`/shares`) that lists all shared users, allows adding a new user by email, and supports removing a shared user with a confirmation dialog.
- New **API endpoints** for global share management (`POST /api/v1/shares`, `DELETE /api/v1/shares/{userId}`).
- **Offline mode**: the Shared Users card is hidden entirely since sharing requires a backend.

## User Stories

1. As a ledger12 user, I want to see how many people have access to my books from the Settings page, so that I can quickly check my sharing status without navigating away.
2. As a ledger12 user, I want to click a button on the Settings page to open a dedicated share management page, so that I can manage shares in a focused UI.
3. As a ledger12 user, I want to see a list of all users I have shared my books with on the Share page, so that I can review who has access.
4. As a ledger12 user, I want to enter another user's email and click "Add Share" to grant them edit access to all my books, so that they can begin collaborating with me immediately.
5. As a ledger12 user, I want to see inline error feedback if I try to share with an email that doesn't match any registered user, so that I know my action failed and why.
6. As a ledger12 user, I want to see inline error feedback if I try to share with my own email, so that I understand sharing with myself is not allowed.
7. As a ledger12 user, I want to see inline error feedback if I try to share with someone who already has access, so that I don't create duplicate shares.
8. As a ledger12 user, I want to remove a user's access by clicking a remove button next to their email, so that I can revoke their access when it's no longer needed.
9. As a ledger12 user, I want a confirmation dialog before a user is removed, so that I don't accidentally revoke access.
10. As a ledger12 user, I want the email input to clear after a successful add, so that I can add another user without manually clearing the field.
11. As a ledger12 user, I want the user list to update immediately after adding or removing a share, so that the UI stays in sync with the backend state.
12. As a ledger12 user who shares with someone, I want that person to have access to any new book I create, so that I don't have to remember to re-share each new book.
13. As a ledger12 user in offline mode, I want the Shared Users section to be hidden from Settings, so that I am not shown a feature that requires a network connection.
14. As a developer, I want the global share endpoints documented in API.md, so that the API contract is the single source of truth.
15. As a developer, I want MSW handlers for the new endpoints, so that frontend development and testing can proceed independently of the backend.
16. As a developer, I want tests for the new service functions, store actions, and SharePage component, so that regressions are caught early.

## Implementation Decisions

### Data model

- Sharing is **global** — adding a user grants `edit` permission on **all books owned by the sharer** (current and future).
- The shared entity is a registered user (identified by userId), not a raw email. If the email doesn't match a registered user, the server returns 404.
- Only outgoing shares (sharer → sharee) are visible on the Share page. Incoming shares (where another user shared with the current user) are not displayed or managed here — that belongs to a future per-book sharing feature.

### API contract

Add the following endpoints to API.md:

**POST /api/v1/shares**

Creates a global share: adds the user with `edit` permission to every book the current user owns (existing books at the time of the call, and implicitly all future books).

| Field | Required | Description |
|-------|----------|-------------|
| `email` | yes | Email of the registered user to share with |

**Response 201:**
```json
{
  "data": {
    "userId": "usr_2",
    "email": "friend@example.com",
    "affectedBooks": 3
  }
}
```

**Errors:**
- `400` — Cannot share with yourself
- `404` — User not found (email doesn't match any registered user)
- `409` — Already shared

**DELETE /api/v1/shares/{userId}**

Removes a user from every book the current user owns.

**Response 200:**
```json
{
  "data": {
    "removed": true,
    "affectedBooks": 3
  }
}
```

**Errors:**
- `404` — Share not found (the user is not currently shared)

**Notes:**

- The share is always `edit` permission. `view` permission will be added at a later stage when per-book sharing is implemented.
- "All books" means the set of books where `ownerId` equals the current user's ID. Books the user was shared into by someone else are not affected.
- When a new book is created, all currently shared users receive `edit` access to it automatically (server-side logic, not a frontend concern).

### Modules

**API.md** — Document the two new global share endpoints, the error responses, and the "share all books" semantics.

**`src/types/api.ts`** — Reuse the existing `ShareResponse` type (`{ userId, permission }`). Optionally add a `GlobalShareResponse` type that includes `affectedBooks` count, though the frontend mostly ignores this count.

**`src/services/booksService.ts`** — Add two new exported functions:

- `addGlobalShare(email: string): Promise<ShareResponse>` — posts to `POST /api/v1/shares`.
- `removeGlobalShare(userId: string): Promise<void>` — deletes `DELETE /api/v1/shares/{userId}`.

These sit alongside the existing per-book share functions (`addShare`, `updateShare`, `removeShare`).

**`src/store/useUsersStore.ts`** — Extend with:

- `addGlobalShare(email: string): Promise<void>` — calls `addGlobalShare` from the service, then calls `fetchUsers` to refresh the list.
- `removeGlobalShare(userId: string): Promise<void>` — calls `removeGlobalShare` from the service, then calls `fetchUsers` to refresh the list.
- The store already holds `users` (fetched during app initialization via `initializeApp`). The Share page derives the "shared users" list by filtering out the current user's own ID from the `users` array. This is sufficient because the current `GET /api/v1/users` endpoint returns all collaborators, and the only collaborators in the system at this stage are outgoing shares.

**`src/mocks/handlers.ts`** — Add MSW handlers:

- `POST /api/v1/shares` — validate email, find user, check not self, check not duplicate, propagate the share to all owned books by adding to `book.sharedWith` array for each book where `book.ownerId === currentUser.id`.
- `DELETE /api/v1/shares/{userId}` — find the user in all owned books' `sharedWith` arrays and remove them.
- The existing `GET /api/v1/users` handler already returns the correct set (current user + collaborators) and requires no changes.

**`src/pages/shares/SharePage.tsx`** — New route page with:

- Title: "Shared Users"
- A list of currently shared users, each row showing email + a "Remove" button.
- A confirmation dialog (using shadcn components) when remove is clicked: "Are you sure you want to remove {email} from all your books? They will lose access immediately." with Cancel/Remove buttons.
- An email input field + "Add Share" button at the bottom.
- Inline error text displayed near the form when add fails.
- Input clears after successful add.
- Empty state: "No shared users yet. Invite someone by entering their email below."
- Reads from `useUsersStore`, derives shared users by filtering out the current user.

**`src/pages/settings/SettingsPage.tsx`** — Add a new Card:

- Title: "Shared Users"
- Description: "{count} user(s) have access to your books" (reads count from `useUsersStore.users.length - 1`, subtracting the current user).
- Button: "Manage Shares…" navigates to `/shares`.
- The entire card is conditionally rendered only when `!isOfflineMode()`.

**`src/routes/index.tsx`** — Add route: `{ path: "shares", element: <SharePage /> }`.

### Architecture notes

- The existing `useUsersStore` is fetched during app initialization in `initializeApp()`. The Settings page can read from this store without an additional fetch.
- The Share page may want to fetch fresh data on mount to ensure accuracy, since the store might be stale. A `React.useEffect` calling `useUsersStore.getState().fetchUsers()` on mount is appropriate.
- Shared users list is derived by: `users.filter(u => u.id !== currentUserId)`.
- No new feature folder is needed — the SharePage lives in `pages/shares/` and the service/store logic is thin enough to live in existing modules.

## Out of Scope

- **Per-book sharing** — this feature is global only. Per-book sharing with `view`/`edit` permissions is deferred.
- **View permission** — all shares are `edit` for now. The `view` permission will be introduced with per-book sharing.
- **Sharing notifications** — the user being shared with is not notified. They simply gain access on next page load.
- **Accept/decline flow** — shares are immediate with no confirmation from the sharee.
- **Incoming share management** — the Share page only shows outgoing shares (users the current user shared with). Managing incoming shares (books shared with the current user by others) is out of scope.
- **Email invitation for unregistered users** — only registered users can be shared with. Sending invitations to unregistered emails is deferred.
- **Bulk operations** — no "remove all" or "select multiple" functionality.
- **Audit log** — no record of who was shared, when, or by whom.
- **Offline mode** — the feature is hidden in offline mode entirely.
- **Keybindings or accessibility shortcuts** — default keyboard navigation only.

## Further Notes

- The existing `GET /api/v1/users` endpoint is used as-is to list shared users. It returns the current user plus all collaborators. The Share page filters out the current user.
- At the moment, the only way a user appears as a collaborator is via an outgoing share, so the `GET /api/v1/users` response is functionally equivalent to "users I have shared with." In the future, when incoming shares exist, a direction filter or separate endpoint may be needed.
- The mock handlers simulate all owned books being updated. This is faithful to the server contract and useful for manual testing.
- The `affectedBooks` field in responses is informational only — the frontend does not display it but may use it for logging or debugging in the future.
