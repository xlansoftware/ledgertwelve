# PRD 009 — Monthly Insight Page & Refactoring

## Problem Statement

The insight pages currently only support a **daily** view (`/insight/daily`) with pie chart, area chart, and selectable day list. Users cannot see their spending patterns at a monthly granularity — month-to-date totals, running balance across the year, or a per-month breakdown of expenses.

Building a monthly view by duplicating the daily components would lead to code duplication and maintenance burden. The daily list and area chart are conceptually reusable across any time granularity, but are currently coupled to day-level date formatting and labels.

## Solution

Introduce a **Monthly Insight** page (`/insight/monthly`) parallel to the daily page, with:

1. **Pie chart** — category breakdown for a selected month (defaults to current month).
2. **Area chart** — year-to-date running balance from January through December, with projection for remaining months.
3. **Selectable month list** — January through current month, with net amount per month. Tapping a month switches the pie chart.

Extract the shared visualization logic into generic components (`PeriodList`, `PeriodAreaChart`) that are granularity-agnostic. Keep the existing daily components (`DailyList`, `DailyAreaChart`) as thin wrappers that apply day-specific formatting. Add corresponding monthly wrappers (`MonthlyList`, `MonthlyAreaChart`).

## User Stories

1. As a user, I want to see the current month's expenses broken down by category in a pie chart, so that I can understand where my money went this month.
2. As a user, I want to tap a month in the list and see the pie chart update to show that month's category breakdown, so that I can explore past months' spending.
3. As a user, I want to see an accumulated area chart of my running balance from January through the current month, so that I can see my year-to-date financial trajectory.
4. As a user, I want a dotted projection line extending from the current month through December, so that I can anticipate where my balance is heading by year-end.
5. As a user, I want the current month to be labeled "This Month" and be selected by default, so that I know which month the pie chart represents.
6. As a user, I want to see a list of months from January through the current month with each month's net total, so that I can quickly scan which months had the most activity.
7. As a user, I want to click on the area chart to select a month and see the pie chart update accordingly, mirroring the daily chart's click-to-select behavior.
8. As a user, I want the selected month to be visually highlighted in the list (background highlight + left border accent), so that I can easily identify which month's data is shown in the pie.
9. As a user, I want each section to have its own loading state (skeleton), so that the page structure appears as data loads.
10. As a user, I want partial errors — if one section fails, the others should still render, so that a single API failure doesn't block the entire page.
11. As a user, I want the pie chart to keep showing its current data while a newly selected month's data is loading, so that I don't experience flicker.
12. As a user, I want the selected month in the list to show a loading indicator while its category data is being fetched, so that I know the app is responding.

## Implementation Decisions

### API Contract — New Endpoint

**`GET /api/v1/reports/monthly?from=&to=`**

Returns monthly net amounts for the Main book within the given date range.

```json
{
  "data": [
    { "period": "2026-01", "amount": -1234.50 },
    { "period": "2026-02", "amount": 2100.00 }
  ]
}
```

- `from` / `to` — ISO 8601 date strings. Both required.
- `period` — `"YYYY-MM"` string (date only, no time component).
- `amount` — net amount for that month (negative = net expense, positive = net income).
- Sorted ascending by period.
- Months with no transactions are omitted from the response.
- Uses Main book only (same as other reporting endpoints).

### DTO — `MonthlyReportRow`

```typescript
export interface MonthlyReportRow {
  period: string   // "YYYY-MM"
  amount: number
}
```

### Modules Built

#### 1. `PeriodList` — Generic Selectable Period List

A deep module encapsulating the list rendering logic. Takes:

```typescript
interface PeriodListProps {
  periods: { period: string; amount: number }[]
  selectedPeriod: string | null    // null = "current" (today / this month)
  currentPeriod: string            // ISO period string for "current" detection
  labelForCurrent: string          // e.g. "Today" or "This Month"
  isLoadingPeriod: boolean
  isLoadingList: boolean
  error: string | null
  onSelect: (period: string | null) => void
  formatLabel: (period: string) => string
}
```

Renders:
- Loading state: skeleton rows.
- Error state: error banner.
- List: periods sorted newest-first, current period highlighted, selected period styled with `bg-accent` + left border accent, loading spinner on the selected row while fetching, net amount colored (negative = red, positive = green, zero = muted).

The component itself has no awareness of days vs. months — it receives all formatting from props.

#### 2. `PeriodAreaChart` — Generic Area Chart

A deep module encapsulating the accumulated area chart with projection. Takes:

```typescript
interface PeriodAreaChartProps {
  data: (AccumulatedRow | ProjectedRow)[]
  isLoading: boolean
  error: string | null
  selectedPeriod?: string | null
  onSelectPeriod?: (period: string) => void
  title: string
  formatLabel: (period: string) => string        // X-axis tick labels
  formatTooltipLabel: (period: string) => string  // Tooltip date/period label
}
```

Renders the same recharts area chart structure as the current `DailyAreaChart` — historical area with gradient, projected dotted line, tooltip showing daily/monthly delta and balance, click-to-select behavior.

No awareness of day vs. month granularity — all date/period formatting comes from props.

#### 3. `InsightMonthlyPage` — Monthly Page Orchestrator

Uses `useMonthlyInsight` hook. Renders three sections stacked vertically:
1. `InsightComponent` (reused) — pie chart for selected month.
2. `MonthlyAreaChart` (wraps `PeriodAreaChart`) — year-to-date running balance + projection.
3. `MonthlyList` (wraps `PeriodList`) — month list.

No business logic in the component.

#### 4. `useMonthlyInsight` Hook

Manages all state, data fetching, and month selection for the monthly page. Parallel structure to `useDailyInsight`.

State:
- Monthly report data (Jan → current month) fetched via `getMonthlyReport`
- Opening balance as of Dec 31 previous year, fetched via `getBookStats`
- Category breakdown for selected month, fetched via `getCategoryReport`
- Selected month (`string | null`, where null = current month)

Accumulation:
- `computeAccumulation(monthlyRows, openingBalance)` — seeded with balance as of Dec 31
- `computeProjection(accumulated, M)` — M = 12 - currentMonth (projected forward to December)

Data flow:
1. **Page load**: three parallel fetches — `getMonthlyReport({ from: Jan 1, to: next month })`, `getBookStats("book_main", { asOf: Dec 31 })`, `getCategoryReport({ from: monthStart, to: monthEnd })`.
2. **Month tap**: `selectMonth(period)` fires `getCategoryReport({ from: monthStart, to: monthEnd })` for that month. Pie stays on previous data until fetch resolves.
3. Accumulation and projection computed from monthly report data, seeded with opening balance.

### Constants

- Default range: January 1 of current year → first day of next month.
- Projection: `M = 12 - currentMonth` (remaining months in the year).
- Opening balance `asOf`: December 31 of previous year.
- Book ID: `"book_main"` (hardcoded, consistent with reporting endpoints).

### Graceful Degradation

- If `getBookStats` fails → opening balance falls back to 0, accumulation starts from zero.
- If `getMonthlyReport` fails → area chart and list show error, pie still works if category data loaded.
- If `getCategoryReport` fails → pie shows error, area chart and list still work.

## Out of Scope

- Navigation between `/insight/daily` and `/insight/monthly` — no tabs or links added. Routes accessible via direct URL only.
- Making the book ID configurable (book selector). Hardcoded to `"book_main"`.
- Making the year range configurable (e.g., showing previous years). Always year-to-date.
- Any changes to the existing `InisghtPage.tsx` (monthly category pie only) or `InsightDailyPage`.
- Persisting selected month across navigations.
- Changing the projection algorithm.
- Adding tests for the generic `PeriodList` or `PeriodAreaChart` in isolation is noted as desirable but listed as in-scope subject to time.

## Further Notes

- The `computeAccumulation` function accepts `"YYYY-MM"` period strings in the `date` field without modification — the strings sort correctly lexicographically and display as labels. No date parsing needed for accumulation logic.
- The monthly area chart's X-axis labels will show abbreviated month names (`"Jan"`, `"Feb"`, ... `"Dec"`) using `date-fns` `format(date, "MMM")`.
- Projection for the monthly view covers remaining calendar months (e.g., Jul through Dec if we're in June). This is computed as `12 - currentMonth` where months are 1-indexed (January = 1).
- Opening balance is fetched as of December 31 of the previous year, using the existing `getBookStats("book_main", { asOf: "2025-12-31" })` endpoint. This gives the net balance at the start of the current year.
- The MSW mock for `GET /api/v1/reports/monthly` groups transactions by `YYYY-MM` format string, sums amounts, and returns sorted ascending. Same pattern as the existing daily and totals mocks.
- The `InsightComponent` (pie chart) is reused as-is — it already handles the `data`/`altData` split for expenses and income.
