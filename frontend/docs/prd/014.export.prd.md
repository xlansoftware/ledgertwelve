# Export Data

## Problem Statement

The app has no way for users to get their data out — no export functionality exists beyond the current hardcoded mock for transactions to CSV/XLSX with a date range. Users cannot back up their books, categories, and transactions in JSON, nor can they generate reports (daily/monthly/yearly balances) as CSV or XLSX for external analysis. The current export API is too narrow: it requires date ranges (`from`/`to`) and only supports CSV/XLSX for transactions. There is no way to export categories, book metadata, or full backups. The feature is also unavailable in offline mode.

## Solution

An export page at `/export` where users select a format (CSV, XLSX, JSON), a content type (categories, transactions, books, reports with sub-selections, or full backup), and an optional book picker for transaction exports. Clicking Export starts an asynchronous job that, when complete, lets the user download the generated file.

The Settings page gains two navigation buttons: "Export Data…" and "Backup…" that navigate to the new export page (the latter pre-selects backup mode via a query parameter).

The backend API is redesigned to remove date range parameters, add a `contentType` enum covering all exportable data types, and support JSON as a first-class format. Backup mode (JSON only) exports all books, categories, and transactions. Job polling adds a `"failed"` status for error reporting.

Offline mode is supported via the existing service factory pattern: an `OfflineExportsService` generates files from IndexedDB data, and an `OnlineExportsService` delegates to the API. Both implement the same `IExportsService` interface.

## User Stories

1. As a user, I want to navigate to an export page from Settings, so that I can export my data in a format I choose.

2. As a user, I want to select CSV, XLSX, or JSON as the export format, so that I can get data in the format my tools need.

3. As a user, I want to export my categories to CSV/XLSX/JSON, so that I can review or edit my category list in a spreadsheet or other tool.

4. As a user, I want to export all transactions (with all fields) for a specific book to CSV/XLSX/JSON, so that I can analyze my spending offline.

5. As a user, I want to select which book to export transactions from, so that I don't mix data from different books.

6. As a user, the book picker for transactions should default to my current book, so that I don't have to search for it every time.

7. As a user, I want to export my book metadata (name, currency, status) to CSV/XLSX/JSON, so that I can keep an inventory of my books outside the app.

8. As a user, I want to generate a daily balance report (total or per-category) as CSV or XLSX, so that I can track daily cash flow in a spreadsheet.

9. As a user, I want to generate a monthly balance report (total or per-category) as CSV or XLSX, so that I can track monthly trends.

10. As a user, I want to generate a yearly balance report (total or per-category) as CSV or XLSX, so that I can do annual financial analysis.

11. As a user, I want reports to use the Main book only, so that the financial summaries match what the app's report pages show.

12. As a user, I want a "Backup" option that exports all my books, categories, and transactions as a single JSON file, so that I can restore my data later if needed.

13. As a user selecting Backup, I want the format selector to be hidden and JSON assumed, so that I don't make a wrong format choice for backup data.

14. As a user selecting Backup, I want all other controls (book picker, content type, report options) to be hidden, so that the form is dead simple.

15. As a user, I want to click a single "Export" button to start the export, so that the process is straightforward.

16. As a user, I want to see the export job progress (pending → processing → completed), so that I know the export is working.

17. As a user, I want to see a download button appear when the export is complete, so that I can save the file to my device.

18. As a user, I want the download to have a descriptive filename based on content type (e.g., `transactions-book_main-2026-06-24.csv`), so that I can distinguish multiple exports.

19. As a user, I want to see an error message if the export fails, so that I know something went wrong and can try again.

20. As a user, I want to trigger a full backup directly from Settings with one click, so that I can quickly back up my data without navigating the full form.

21. As a user, I want a quick "Backup…" button in Settings that navigates to `/export?preset=backup`, so that the export page opens already configured for backup.

22. As an offline user, I want to export my data to CSV or JSON from local storage, so that I don't lose the ability to extract my data when offline.

23. As an offline user, I want the export process to feel identical to online mode (same job-based flow), so that the experience is consistent regardless of connectivity.

24. As a developer, I want the export feature to be implemented via a service interface with separate online and offline implementations, so that the UI code does not need to know about connectivity.

25. As a developer, I want both the online and offline export services to be tested via the same interface contract, so that I can verify both implementations behave identically.

## Implementation Decisions

### API Contract — `POST /api/v1/exports`

The existing export endpoint is redesigned. `from` and `to` are removed entirely. A new `contentType` field selects the export type. `format` now also accepts `"json"`. The `bookId` field is only required when `contentType = "transactions"`.

**Request body:**

```json
{
  "format": "csv",
  "contentType": "transactions",
  "bookId": "book_main"
}
```

**Validation rules:**

| Field | Required | Valid values | Notes |
|---|---|---|---|
| `format` | yes | `"csv"`, `"xlsx"`, `"json"` | Ignored when `contentType = "backup"` — server forces `"json"` |
| `contentType` | yes | `"categories"`, `"transactions"`, `"books"`, `"report-daily-total"`, `"report-daily-per-category"`, `"report-monthly-total"`, `"report-monthly-per-category"`, `"report-yearly-total"`, `"report-yearly-per-category"`, `"backup"` | |
| `bookId` | conditional | valid book ID | Required when `contentType = "transactions"`. Ignored when `contentType = "backup"`. Reports use Main book only. |

**Response (201):**

```json
{
  "data": {
    "jobId": "exp_123",
    "status": "pending"
  }
}
```

### API Contract — `GET /api/v1/exports/{jobId}`

The job status enum gains `"failed"` with an optional `errorMessage`.

```json
// Processing
{ "data": { "jobId": "exp_123", "status": "processing" } }

// Completed
{ "data": { "jobId": "exp_123", "status": "completed", "downloadUrl": "/api/v1/exports/exp_123/download" } }

// Failed
{ "data": { "jobId": "exp_123", "status": "failed", "errorMessage": "Failed to generate XLSX: insufficient memory" } }
```

### Download filenames

Filenames set via `Content-Disposition` header, based on content type:

| contentType | Example filename |
|---|---|
| `categories` | `categories-2026-06-24.csv` |
| `transactions` | `transactions-book_main-2026-06-24.xlsx` |
| `books` | `books-2026-06-24.json` |
| `report-daily-total` | `report-daily-total-2026-06-24.csv` |
| `backup` | `ledger12-backup-2026-06-24.json` |

### JSON backup shape

```json
{
  "exportedAt": "2026-06-24T12:00:00Z",
  "version": 1,
  "books": [ ... ],
  "categories": [ ... ],
  "transactions": [ ... ]
}
```

Does not include user identities or other non-user-owned data.

### Route structure

```
/export              → ExportPage (child of App layout, under Header)
```

### Settings page integration

A new card at the bottom of the Settings page (after the Theme card) with two buttons:

- **"Export Data…"** — navigates to `/export`
- **"Backup…"** — navigates to `/export?preset=backup`

### Export page layout

- **Format selector**: Radio buttons or segmented control for CSV, XLSX, JSON
- **Content type selector**: Radio buttons for Categories, Transactions, Books, Reports, Backup
- **Book picker** (shown only when Transactions selected): Dropdown populated from `useBooksStore`, defaulting to `currentBook`
- **Report sub-options** (shown only when Reports selected):
  - Period: Daily, Monthly, Yearly (radio buttons)
  - Group by: Total, Per category (radio buttons)
- **Export button**: Primary action button
- **Backup mode**: When Backup is selected, the format selector, book picker, and report options are hidden. Only the Export button remains, with a note: "Full backup — all books, categories, and transactions as JSON."
- **Progress area**: After clicking Export, the form is replaced or supplemented with a progress indicator showing the job state. When completed, a download button is revealed.

### Service factory architecture

Following the existing offline/online service pattern, a new `IExportsService` interface is introduced:

```ts
interface IExportsService {
  createExport(req: CreateExportRequest): Promise<CreateExportResponse>
  getExportJob(jobId: string): Promise<ExportJobDto>
  downloadExport(jobId: string): Promise<Blob>
}
```

- **`OnlineExportsService`** — thin wrapper delegating to the API (replaces/updates the current `exportsService.ts`)
- **`OfflineExportsService`** — generates files from IndexedDB data in-memory, stores as blob URLs, resolves jobs immediately to `"completed"` with a `blob:` download URL. Supports CSV and JSON natively. XLSX generation in offline mode requires an additional library or falls back to CSV with a note.
- Both are registered in the `ServiceFactory` and the factory is added to `setFactory`/`createOnlineFactory`/`createOfflineFactory`

## Testing Decisions

Tests should verify behavior, not implementation details. Test the export service interface contract — both online and offline implementations must satisfy the same contract.

- **OnlineExportsService** — test via MSW that `createExport` sends the correct request, that job polling transitions through states, that failed jobs return an error, and that `downloadExport` returns a Blob. Prior art: `exportsService.test.ts` (existing tests).
- **OfflineExportsService** — test that `createExport` for each content type produces a successful job, that job polling returns `completed`, that `downloadExport` returns a non-empty Blob, and that backup JSON contains the expected top-level fields. Do not mock IndexedDB — use a real in-memory IndexedDB or the existing DB mock infrastructure. Prior art: offline service tests in `features/offline/offline/__tests__/`.
- Both services must be tested via the same `IExportsService` interface. The test suite should be parameterized to run the same behavioral tests against both implementations.
- No tests for `ExportPage`, `useExport`, or `ExportProgress` — these are UI wiring and state orchestration components that are exercised by the integration tests above and manual QA.

## Out of Scope

- Import/restore functionality — JSON backup is export-only. Restore is a future feature.
- XLSX generation without a library in offline mode — if no XLSX library is available, offline XLSX exports will either return an error or fall back to CSV. This is a build-time dependency decision outside this PRD.
- Pagination of large exports — the assumption is that exports of all data for a single user fit in memory. Large-scale optimization is out of scope.
- Email exports or sharing — the feature is download-only.
- Scheduled/automated exports — all exports are user-initiated.
- Export history — the page does not show a list of past exports beyond the current job. Each export starts fresh.
- Multi-language/i18n — no translations for this feature.

## Further Notes

- The `contentType` enum is intentionally flat on the API side. The UI uses a two-level selection (pick "Reports" then pick period + group-by) and maps it to a flat value. This keeps the API contract simple.
- When `contentType = "backup"`, the server ignores `format` and `bookId` entirely — no validation errors for unexpected values. This allows the client to send the same request body shape regardless of content type.
- The existing `from`/`to` date range removal means all exports are full-data exports. If a scoped date range is needed later, it can be added back as an optional filter rather than a required parameter.
- The offline export service may need a note about XLSX support: if no client-side XLSX library is available, the user can select CSV or JSON instead. The format selector in offline mode should either hide XLSX or show a disabled option with a tooltip explanation.
