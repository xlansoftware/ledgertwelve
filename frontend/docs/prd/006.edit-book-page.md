# PRD 006 — Edit Book Page

## Problem Statement

The book list page (`/books`) only allows selecting a book as the "current" book and creating new books. There is no way to:

- Edit a book's name or currency after creation
- View book statistics (creation date, transaction count, total sum)
- Close or reopen a book
- Delete a book

Users must live with whatever they set at creation time, and there is no overview of a book's aggregate data.

## Solution

Introduce an **Edit Book** page (`/edit-book/:bookId`) that consolidates book management in one place: editable metadata, read-only statistics, status controls (close/reopen), and a delete action.

The new page is reached by clicking an **Edit** button added to each book card on the books page. The button is always shown, regardless of user permission — the API enforces authorization.

## User Stories

1. As a user, I want an "Edit" button on each book card on the books page, so that I can navigate to the edit page for that book.
2. As a user, I want to change a book's name and currency after creation, so that I can correct mistakes or update book details.
3. As a user, I want to see when a book was created, so that I can understand its age.
4. As a user, I want to see how many transactions a book contains, so that I can judge how active the book is.
5. As a user, I want to see the total (net) sum of all transactions in a book, so that I can quickly assess its balance.
6. As a user, I want to close a book, so that I can finalise its balances and move its net balance to the Main book.
7. As a user closing a book, I want to pick a category for the balancing transaction, so that the closing entry is properly categorised.
8. As a user, I want to reopen a closed book, so that I can continue using it.
9. As a user, I want to delete a book, so that I can remove books I no longer need.
10. As a user deleting a book, I want a confirmation dialog, so that I don't accidentally lose data.
11. As a user, I want informative error messages when actions fail (e.g. cannot delete Main book, cannot delete non-empty book), so that I understand why the operation was rejected.
12. As a user, I want to see a loading state while the edit page is fetching data, so that I know the page is working.
13. As a user, I want to see an error state when the book cannot be loaded (not found, not visible), so that I know something went wrong.

## Implementation Decisions

### API Contract — New Endpoint

**`GET /api/v1/books/{bookId}/stats`**

Returns lightweight read-only statistics for a book.

```json
{
  "data": {
    "transactionCount": 142,
    "totalSum": -4523.50
  }
}
```

- `transactionCount` — number of non-closing transactions in the book.
- `totalSum` — net sum of all non-closing transaction amounts.
- Requires the book to be visible to the current user (owned or shared).
- Returns `404` if the book is not found or not visible.

### DTO — `BookStatsDto`

```typescript
export interface BookStatsDto {
  transactionCount: number
  totalSum: number
}
```

### Modules Built / Modified

1. **`API.md`** — add the `/api/v1/books/{bookId}/stats` endpoint definition.
2. **`src/types/dto.ts`** — add `BookStatsDto`; export from `src/types/index.ts`.
3. **`src/services/booksService.ts`** — add `getBookStats(bookId)` function; export from `src/services/index.ts`.
4. **`src/mocks/handlers.ts`** — add MSW handler for `GET /api/v1/books/{bookId}/stats` that computes stats from the in-memory transaction store.
5. **`src/pages/books/EditBookPage.tsx`** — new page component for the edit page.
6. **`src/pages/books/CloseBookDialog.tsx`** — new dialog component for the close-book workflow. Encapsulates the category combobox, close confirmation, and API call. Simple interface: `{ bookId, bookName, open, onOpenChange, onSuccess }`.
7. **`src/pages/books/BookPage.tsx`** — add "Edit" button in each book card, navigating to `/edit-book/:bookId`.
8. **`src/routes/index.tsx`** — add route `/edit-book/:bookId` → `EditBookPage`.

### Page Layout (`EditBookPage`)

The page has sections from top to bottom:

1. **Header** — "Edit Book" title, [Cancel] and [Save] buttons.
2. **Name / Currency form** — two text inputs pre-populated from the book data. Cancel navigates back to `/books`. Save calls `PUT /api/v1/books/{bookId}` and shows a success toast on completion.
3. **Statistics** — read-only display of:
   - Created date (formatted locale string)
   - Transaction count
   - Total sum (formatted as currency)
4. **Status section** — current status badge ("Open" / "Closed") + action button:
   - If **open**: "Close book" button → opens `CloseBookDialog`
   - If **closed**: "Reopen book" button → confirmation dialog → calls `POST /api/v1/books/{bookId}/reopen`
5. **Danger zone** — "Delete book" button → confirmation dialog → calls `DELETE /api/v1/books/{bookId}` → navigates to `/books` on success.

All error responses from the API are surfaced as error banners or toasts.

### Data Flow

- On mount: fetch book detail (`GET /api/v1/books/{bookId}`) and stats (`GET /api/v1/books/{bookId}/stats`) in parallel.
- Pre-populate the form from the fetched book detail.
- All mutations (save, close, reopen, delete) go through the store (`useBooksStore`) for optimistic updates and error handling.
- Stats are page-local state (no global store).

### Store Changes

None. The existing `useBooksStore` actions (`updateBook`, `deleteBook`, `closeBook`, `reopenBook`) are sufficient. Stats are fetched directly via the service and held in component state.

### State Handling

- **Loading**: skeleton placeholders for form fields and statistics.
- **Error (fetch failure)**: error message with a "Back to books" button.
- **Save errors**: banner above the form.
- **Action errors (delete, close, reopen)**: toast notifications.

### Routing

```typescript
{
  path: "edit-book/:bookId",
  element: <EditBookPage />,
}
```

- Test external behavior, not implementation details.
- Use MSW for API mocking (already set up in the project).
- Prefer user-visible assertions (element content, button labels, toasts, navigation).

## Out of Scope

- Sharing management (add/edit/remove shares) — already handled by the existing book detail in the store, but not surfaced on this page.
- Book-level transaction search or listing — the book's transactions are managed from the history page.
- Multi-book statistics or aggregate reporting.
- Bulk operations on books.
- Sorting, filtering, or pagination of books on the edit page.

## Further Notes

- The "Edit" button is always rendered on every book card, even for view-only shared books. The API will reject unauthorized actions — errors are surfaced as toasts.
- The close-book dialog must show all user categories alphabetically, using the existing category combobox pattern from the add-transaction page.
- The delete confirmation dialog uses the existing `ConfirmDialogProvider` already wired up in the app.
- All formatting of dates and currency amounts should use the browser's built-in `Intl` APIs for locale-aware display.
