# Login Page

## Problem Statement

The frontend currently hardcodes a login call in the bootstrap phase (`login({ email: 'john@example.com', password: 'secret-password' })`), with no way for users to authenticate interactively, switch accounts, or log out. A user who opens the app has no choice but to be automatically signed in as a single pre-configured test user.

## Solution

A login page where users can sign in with their email and password, or start using the app with locally stored data (local mode). The app checks for an existing session on startup and routes unauthenticated users to the login page rather than crashing with an error screen. A settings card shows the current authentication status and provides a logout action.

## User Stories

1. As a user opening the app for the first time, I want to see a login page, so that I can authenticate before using the application.

2. As a returning user, I want the app to check if I already have a valid session, so that I can skip the login page and go straight to the app.

3. As a user with credentials, I want to enter my email and password and click a sign-in button, so that I can authenticate with the backend.

4. As a user who enters wrong credentials, I want to see an inline error message, so that I know my login attempt failed.

5. As a user who wants to try the app without setting up an account, I want a "Use App Offline" button, so that I can access the app with demo data immediately.

6. As an authenticated user, I want to see my authentication status in the settings page, so that I know which account I'm logged in with and whether I'm in online or local mode.

7. As an authenticated user, I want to click a logout button in settings, so that I can end my session and return to the login page to sign in as a different user.

8. As a user on the login page who is already authenticated, I want to be allowed to sign in again as a different user, so that I can switch accounts without manually logging out first.

## Implementation Decisions

### Architecture overview

The authentication system is split into three modules:

- **Auth store** — A Zustand store holding the authentication state. Single source of truth for whether the user is authenticated, in local mode, or unauthenticated. Exposes `login()`, `logout()`, and `setLocalMode()` actions that update the state and persist nothing (session-based).

- **Login page** — A standalone page (no Header, no app chrome) with a centered card containing email input, password input, a "Sign In" button, and a "Use App Offline" button. Handles form submission and error display.

- **Auth guard** — A check in `App.tsx` that redirects unauthenticated users to `/login`. Reads the auth store and returns a `<Navigate to="/login" replace />` when `status === 'unauthenticated'`.

### Route structure

The login route lives as a sibling to the root layout, not as a child, so it renders without the Header.

```
/login             → LoginPage (standalone)
/                   → App (Header + Outlet)
  / (index)         → AddPage
  /history          → HistoryPage
  /books            → BookPage
  /books/new        → CreateBookPage
  /edit-book/:id    → EditBookPage
  /edit-transaction/:id → EditTransactionPage
  /settings         → SettingsPage
  /categories       → CategoriesPage
  /insight          → InisghtPage
  /insight/daily    → InsightDailyPage
  /insight/monthly  → InsightMonthlyPage
*                   → NotFoundPage
```

### Bootstrap flow

The `main.tsx` entry point changes from:

1. Show loading spinner
2. Start MSW
3. **Hardcoded login** → error screen if it fails
4. Fetch reference data
5. Render app

To:

1. Show loading spinner
2. Start MSW
3. `whoami()` → **200**: set `authenticated`, fetch reference data, render app. **401**: set `unauthenticated`, render app (router shows login page)
4. Remove hardcoded `login()` call entirely

If MSW start fails, render error screen (unchanged).

### Auth state shape

```ts
// From prototype — the discriminated union for auth state
type AuthState =
  | { status: 'unauthenticated' }
  | { status: 'authenticated'; user: UserSummary }
  | { status: 'local' }            // placeholder for future offline mode
```

The `'local'` status exists now but will be fully implemented in a separate feature. For now, "Use App Offline" sets `authenticated` with the hardcoded demo user, the same as the old bootstrap flow.

### Login page

- Standalone page, no Header, no app chrome
- Uses existing shadcn components: `Card`, `Input`, `Label`, `Button`
- Form fields: email (not username), password
- "Sign In" button calls `login()` from authService
- "Use App Offline" button calls `login({ email: 'john@example.com', password: 'secret-password' })` — the same hardcoded credentials previously in main.tsx
- Shows inline error ("Invalid email or password") on failed login
- On success, navigates to `/`
- If already authenticated, still allows signing in as a different user (no redirect away)

### Settings card

A new card in the existing Settings page showing:

- Current authentication mode: "Online — {email}" or "Local mode"
- A "Logout" button

On logout:
1. Call `POST /api/v1/auth/logout`
2. Set auth store to `unauthenticated`
3. Navigate to `/login`

### API contract — Logout endpoint

New endpoint added to `API.md`:

```
POST /api/v1/auth/logout
- Purpose: End the current authenticated session
- Request: No body
- Response: { data: { success: true } }
- Server clears the session cookie
```

## Testing Decisions

Tests should verify behavior, not implementation details. For auth:

- **authService.test.ts** — already tests `login()` and `whoami()` with MSW. Add tests for `logout()` following the same pattern (import `server`, test success case).
- **LoginPage test** — verify that the form renders, that clicking "Sign In" with empty fields does not call the API, that error messages appear on failure, that navigation happens on success. Use `@testing-library/react` and MSW. Prior art: `AddPage.test.tsx`, `HistoryPage.test.tsx`.
- **useAuthStore test** — verify that `login()` updates state from `unauthenticated` to `authenticated`, that `logout()` sets `unauthenticated`. No MSW needed if login/logout are unit-tested separately — the store actions just call services and update state.
- **Auth guard (App.tsx)** — render the guarded app with different auth store states and assert that unauthenticated users are redirected. Prior art: `BookPage.test.tsx` for router tests.
- **Settings page test** — verify the auth card renders with correct user email and that clicking "Logout" calls the API and redirects. Prior art: `SettingsPage` currently has no test, but follow patterns from `CreateBookPage.test.tsx`.

## Out of Scope

- Offline/local-mode data storage (IndexedDB, localStorage) — the "Use App Offline" button currently just logs in with hardcoded demo credentials. Real offline mode is a future feature.
- Registration / "Create account" flow.
- Password reset / "Forgot password" flow.
- Multi-factor authentication.
- Session timeout or auto-logout.
- OAuth / social login.
- Remember-me / persistent sessions.
- Unit tests for the auth guard in App.tsx (integration-level test, lower priority).

## Further Notes

- The existing `currentUser` export from `mocks/handlers.ts` and `clearAuth()` function already support the logout flow in tests and dev mode.
- The "Use App Offline" button is intentionally a simple hardcoded login for now. Its label and behavior may change when real offline mode is implemented.
- The auth store does not persist to localStorage — session state is lost on full page reload. The `whoami()` call on startup handles session recovery when a server-side session cookie exists.
