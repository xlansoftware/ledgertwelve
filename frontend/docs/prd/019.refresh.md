# Data Refresh Infrastructure

## Problem Statement

The application currently has no centralized mechanism for refreshing data. Key pages like BookPage and HistoryPage rely on the store state that was seeded once during `initializeApp()` (on startup) but never refresh when the user navigates to them. Similarly, after a data-changing operation like import or backup restore, there is no automated way to ensure all affected stores are re-fetched — the user would have to manually reload the page to see fresh data.

Each store already exposes `fetch*` methods and handles optimistic updates for its own create/update/delete operations, but there is no coordination layer that says "refresh everything that was affected by this action."

## Solution

Introduce two exports from a single module:

1. **`refresh(targets)`** — an imperative async function that accepts a set of entity types and calls each store's respective `fetch*` method.
2. **`useRefresh(targets)`** — a React hook that calls `refresh(targets)` automatically on mount, and also returns nothing (pure side-effect hook).

This gives a simple, unified API:
- Pages use `useRefresh({ books: true })` to auto-fetch on mount.
- Callbacks (import complete, restore complete) use `await refresh({ books: true, transactions: true, categories: true })` to trigger a manual refresh.

The stores themselves continue handling their own optimistic updates for individual CRUD operations. The refresh layer only covers scenarios where the caller doesn't know exactly which data changed — page navigation and bulk operations.

## User Stories

1. As a user navigating to the BookPage, I want the book list to be fetched from the server, so that I see up-to-date books and don't rely on stale data from initial app load.

2. As a user navigating to the HistoryPage, I want the transaction list to be fetched from the server, so that I see transactions that may have been added by other users or by import since I last viewed the page.

3. As a user who just completed an import of transactions and categories, I want the corresponding stores to automatically refresh, so that I can immediately see the imported data without manually reloading.

4. As a user who just restored from a backup, I want all stores (books, transactions, categories, users) to refresh, so that the full application state reflects the restored data.

5. As a user who navigates away and back to a page, I want the page to re-fetch its data on mount, so that I always see the latest state without pressing a refresh button.

6. As a developer, I want a single, testable `refresh()` function that I can call from any callback or effect, so that I don't need to import and invoke individual store actions scattered across the codebase.

7. As a developer, I want the hook to have a stable reference (empty dependency array) so that it doesn't cause unnecessary re-fetches when the component re-renders.

8. As a developer, I want the refresh to preserve the current query context for transactions (current book, filter, page), so that refreshing doesn't reset the user's view.

9. As a developer, I want errors during refresh to be handled by the stores themselves (which already store errors in their own state), so that the refresh layer doesn't need its own error handling logic.

10. As a developer, I want the hook to only fire on mount (not on re-renders or dependency changes), so that data fetching is predictable and does not cause infinite loops.

## Implementation Decisions

### Module location

A single file `src/hooks/useRefresh.ts` exports both the hook and the imperative function. This follows the existing pattern of placing hooks in `src/hooks/`. The hook is a cross-cutting concern, not domain-specific, so it does not belong in a feature folder.

### API shape (from prototype)

```ts
type RefreshTargets = {
  books?: boolean
  transactions?: boolean
  categories?: boolean
  users?: boolean
}

// Imperative function — for callbacks (import, restore, etc.)
async function refresh(targets: RefreshTargets): Promise<void>

// Hook — auto-refresh on mount
function useRefresh(targets: RefreshTargets): void
```

### Refresh behavior per entity

| Target | Store method called | Notes |
|--------|-------------------|-------|
| `books` | `useBooksStore.getState().fetchBooks()` | Full reload. Also re-derives `mainBookId` and re-fetches the current book selection. |
| `transactions` | `useTransactionsStore.getState().fetchTransactions(lastParams)` | Preserves the current book, filter, and pagination by using the store's `lastParams`. Empty `lastParams` defaults to no filter (all transactions). |
| `categories` | `useCategoriesStore.getState().fetchCategories()` | Full reload. |
| `users` | `useUsersStore.getState().fetchUsers()` | Full reload. |

### Sequential execution

Within a single `refresh()` call, each entity is refreshed sequentially (one `await` at a time in declaration order). This avoids overloading the network with parallel requests and keeps error isolation simple. No debouncing or cross-call deduplication is implemented — the stores already handle their own stale-response guards.

### Error handling

Errors are silently swallowed at the `refresh()` level. Each store independently catches errors and stores them in its own `error` state field, which UI components already read and display. This makes `refresh()` safe to call from any context (useEffect, event handlers) without requiring the caller to wrap it in try/catch.

### Hook implementation

The `useRefresh()` hook calls `refresh(targets)` inside a `useEffect` with a stable dependency array so it fires exactly once on mount. It does not accept dynamic targets that change over time — the targets are fixed for the lifetime of the component.

### Trigger points

| Location | Call | Rationale |
|----------|------|-----------|
| `BookPage` mount | `useRefresh({ books: true })` | Book list may be stale since app init |
| `HistoryPage` mount | `useRefresh({ transactions: true })` | Transaction list may be stale since last visit |
| After import completes (step === "complete") | `await refresh({ books: true, transactions: true, categories: true })` | Import can affect all three |
| After backup restore (step === "complete" with entityType === "backup") | `await refresh({ books: true, transactions: true, categories: true, users: true })` | Restore replaces entire data set |

### No modifications to existing stores

The stores remain unchanged. They already expose `fetch*` methods that the hook calls via `getState()`. No new `refresh()` actions are added to any store.

### No modifications to existing optimistic updates

Individual CRUD operations (create transaction, edit book, delete category) continue to use their existing optimistic update patterns within each store. The refresh layer is only for scenarios where optimistic updates don't apply (page navigation, bulk operations).

## Testing Decisions

Good tests verify behavior from the caller's perspective, not implementation details. For this feature:

- **`useRefresh.test.ts`** — Unit test the hook and imperative function:
  - `refresh({ books: true })` calls `fetchBooks()` on the books store
  - `refresh({ transactions: true })` calls `fetchTransactions()` with the store's `lastParams`
  - `refresh({ categories: true })` calls `fetchCategories()`
  - `refresh({ users: true })` calls `fetchUsers()`
  - `refresh({})` calls no store methods
  - `refresh({ books: true, transactions: true })` calls both (sequential, order doesn't matter)
  - Errors from individual store fetches do not propagate from `refresh()`
  - `useRefresh({ books: true })` calls `refresh()` once on mount with the correct targets
  - `useRefresh({})` calls `refresh()` with empty targets (no-op)
  - Prior art: existing store tests (`useBooksStore.test.ts`, `useTransactionsStore.test.ts`) that mock the factory and assert that service methods are called

- **No component-level tests** — The hook's behavior is fully covered by the unit test above. Integration with pages (BookPage, HistoryPage) is already tested in their existing component tests, which verify rendering behavior. Adding refresh to those pages is a simple import + call that the unit tests cover.

### Test file location

- `src/hooks/useRefresh.test.ts`

## Out of Scope

- Adding a `refresh()` action to any store — the hook calls existing `fetch*` methods directly.
- Periodic polling or interval-based refresh — only mount-time and manual callbacks.
- WebSocket or real-time push invalidation — the refresh is always pull-based from the UI.
- Loading state or loading indicator at the `refresh()` level — each store already tracks its own `isLoading`.
- Debouncing or batching rapid `refresh()` calls — callers are responsible for calling it judiciously.
- Any changes to the existing import workflow UI — only adding the post-import refresh call.
- Any changes to the existing store optimistic update logic.
- Refreshing individual book detail (`fetchBook`) or individual transaction detail (`fetchTransaction`) — only list endpoints are covered.
- Navigation-based auto-refresh for pages beyond BookPage and HistoryPage — other pages either fetch their own data via custom hooks (insight pages, dashboard) or don't need mount-time refresh.
- Error banners or UI changes for refresh failures — the stores already display errors in the UI.

## Further Notes

- The `initializeApp()` function in `src/lib/init.ts` already fetches books, categories, and users on startup. The refresh feature is complementary — it re-fetches this data on page navigation and after bulk operations, rather than relying on the initial load for the entire session.
- Transactions are the only entity not fetched during `initializeApp()`. They are fetched lazily when the user visits HistoryPage. The refresh hook makes this explicit and systematic.
- The import workflow currently has a `goToHistory()` navigation helper that redirects to `/history`. When that navigation triggers HistoryPage's mount, the `useRefresh({ transactions: true })` hook will automatically re-fetch transactions. The explicit `refresh()` call after import is still needed for books and categories, which may affect other pages the user might navigate to instead of History.
