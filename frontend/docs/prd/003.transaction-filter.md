# PRD: Transaction History — Filtering

## Problem Statement

The History page shows all transactions in the current book with no way to narrow the list. As the transaction count grows (the mock generates 1000+), users have no way to find specific transactions — by date range, category, user, note content, or value range. They must scroll through the entire list or rely on external search, which is time-consuming and error-prone.

## Solution

Add a filter mechanism to the History page. A "Filter" button with a magnifying glass icon sits above the transaction list (top-left). Clicking it opens a responsive filter dialog (centered modal on desktop, bottom sheet on mobile) that exposes multiple filter dimensions: date range, category, user, note search, and value range.

The active filter is stored in the Zustand store and persists across navigation within the app. When a filter is active, the transaction count label updates to "XXXX transactions, filtered". Clicking that label clears the filter and refetches all transactions.

## User Stories

1. As a user viewing my transaction history, I want to see a "Filter" button with a magnifying glass icon above the transaction list, so that I know filtering is available and can activate it.

2. As a user who clicks the Filter button, I want to see a dialog (or bottom sheet on mobile) with all available filter options, so that I can configure which transactions to show.

3. As a user, I want to filter transactions by date range (today, this week, this month, this year, all time, or a custom start/end date), so that I can see transactions from a specific period.

4. As a user, I want to filter transactions by one or more categories, so that I can focus on transactions from specific spending categories.

5. As a user, I want to filter transactions by one or more users (when the book is shared), so that I can see only transactions created by specific collaborators or myself.

6. As a user, I want to filter transactions by a note substring search, so that I can find transactions whose notes contain a specific word or phrase.

7. As a user, I want to filter transactions by a minimum and/or maximum value, so that I can find transactions within a specific amount range.

8. As a user, I want each filter dimension to have its own enable/disable toggle, so that I can easily enable or disable an entire filter category without losing my selections.

9. As a user, I want to see "Apply Filters" and "Cancel" buttons at the bottom of the filter dialog, so that I can commit my selections or discard them.

10. As a user who has applied a filter, I want the transaction count to change to "XXXX transactions, filtered", so that I can see at a glance that the view is filtered.

11. As a user who has applied a filter and wants to return to the full list, I want to click the "filtered" label to clear the filter and see all transactions again.

12. As a user who switches to a different book, I want any active filter to be automatically cleared, so that I don't see stale filter results from the previous book.

13. As a user who applies a filter, I want the transaction list to reset to page 1 and load the first page of filtered results, so that I see filtered results immediately.

14. As a user who reopens the filter dialog after applying a filter, I want the dialog to reflect my currently active filter selections, so that I can see what's applied and modify it.

15. As a mobile user, I want the filter to open as a bottom sheet, so that it's easy to reach on a small screen.

## Implementation Decisions

### State ownership
The active filter is stored in `useTransactionsStore` as a new `currentFilter: FilterRequest` field. This is shared state — it persists when navigating away from and back to the History page.

### Filter application mechanics
The store exposes a `setFilter(filter: FilterRequest)` action that:
1. Saves the filter to `currentFilter`
2. Resolves the `period` field to concrete `from`/`to` dates (client-side resolution)
3. Converts `FilterRequest` to `GetTransactionsParams`
4. Resets pagination (page 1)
5. Calls `fetchTransactions` with the new params

A `clearFilter()` action resets `currentFilter` to defaults and refetches without filter params.

### Filter reset on book switch
When `currentBook` changes in `useBooksStore`, `useTransactionsStore` automatically clears the current filter before fetching transactions for the new book.

### API contract changes
The `GET /api/v1/transactions` endpoint is extended with new query parameters:

- `note` — substring search on transaction notes (case-insensitive contains)
- `minValue` — minimum transaction amount (inclusive)
- `maxValue` — maximum transaction amount (inclusive)
- `category` — repeated param, each value is a category name (OR match)
- `createdBy` — repeated param, each value is a user ID (OR match)

The `from` and `to` params remain unchanged for date range filtering. The `period` field from the UI is resolved client-side — the API never sees it.

### New users endpoint
A new endpoint `GET /api/v1/users` returns all users the current user has ever interacted with — the current user themself plus all collaborators from shared books. Each entry is a `UserSummary` (id, email). This powers the users combo-box in the filter dialog.

### Responsive filter container
A new `FilterDialog` component wraps the existing `Filter` component in either a `Dialog` (desktop, centered modal) or a `Sheet` with `side="bottom"` (mobile, bottom sheet), following the pattern established by `ConvertDialog`. The breakpoint is `(max-width: 768px)` via `useMediaQuery`.

### UI layout
- The filter trigger sits top-left in the History header: a magnifying glass icon (from lucide-react) followed by "Filter" text
- The transaction count label sits top-right — shows "XXXX transactions" normally, "XXXX transactions, filtered" when a filter is active
- When "XXXX transactions, filtered" is clicked, the filter is cleared and data is refetched

### Users data fetching
Users are fetched eagerly when the HistoryPage mounts via `useUsersStore.fetchUsers()`. The users list is passed through `FilterDialog` into `Filter`.

### URL state
The filter is NOT synced to URL search params. It lives only in the Zustand store. Filter state is lost on full page reload.

## Testing Decisions

### Testing philosophy
- Test external behavior, not implementation details
- Use Vitest for unit tests and React Testing Library for component tests
- Mock network requests using MSW
- Prefer user-visible assertions (label text, visibility, clicks)
- Avoid testing internal state of the store directly — test through the UI or through the store's public API

### Modules to test

**1. `useTransactionsStore`** — Test `setFilter` and `clearFilter` actions:
- Applying a filter updates `currentFilter` and triggers a fetch with converted params
- Clearing a filter resets `currentFilter` and triggers a fetch without filter params
- Changing the active book clears the filter

**2. `Filter` component** — Test the filter form:
- Toggling each dimension's switch enables/disables the corresponding inputs
- Selecting categories, users, dates, notes, and values updates the local state
- Clicking "Apply Filters" calls `onApply` with the correct `FilterRequest` shape
- Clicking "Cancel" calls `onClose`

**3. `FilterDialog` component** — Test the responsive wrapper:
- Opens when `open` is true, closes when `open` is false
- Renders the `Filter` component inside
- Uses `Dialog` on desktop viewport and `Sheet` on mobile viewport

**4. `HistoryPage` integration** — Test the full flow:
- Filter button is visible in the header
- Clicking the filter button opens the dialog
- Applying a filter shows "filtered" in the transaction count label
- Clicking the "filtered" label clears the filter and reverts to "transactions"
- Filter is cleared when switching books

**Prior art:** Existing tests like `useTransactionsStore.test.ts`, `transactionsService.test.ts`, and `HistoryPage` patterns serve as prior art for the testing approach.

## Out of Scope

- Persisting filter state across full page reloads (no URL params)
- Server-side pagination of filtered results on `loadMore` — the existing `loadMore` mechanism is sufficient
- Sorting — transactions remain sorted by date descending
- Saving named filter presets
- Exporting only filtered results
- Filtering within the reports or other pages
- Search-as-you-type / debounced instant filtering — filter is applied on explicit "Apply" click

## Further Notes

- The `Filter` component already exists at `src/pages/history/Filter.tsx` and is fully functional — it just needs to be wired into the page via the new `FilterDialog` wrapper
- The `FilterRequest` type in `src/types/api.ts` already contains all the fields needed (`period`, `startDate`, `endDate`, `category`, `user`, `note`, `minValue`, `maxValue`) — no type changes required
- The existing `ConvertDialog` at `src/pages/add/ConvertDialog.tsx` is the template for the responsive Dialog/Sheet pattern
- The existing `useCategoriesStore` is the template for the new `useUsersStore`
