# Closed Books List on BookPage

## Problem Statement

The BookPage currently displays all books — open and closed — as identical cards in a single list. As users close books over time, the page becomes cluttered with inactive books that are no longer useful for day-to-day transactions. There is no visual distinction between active books (where the user can still transact) and archived closed books.

## Solution

The BookPage splits books into two visual sections separated by a horizontal rule:

- **Open books** — displayed as cards (unchanged from current behavior), with Select and Edit buttons.
- **Closed books** — displayed as compact single-row cards below a `<Separator />`, showing the book name, a "Closed" badge, the computed balance (from `getBookStats`), and an Edit button. Closed books are not selectable as the current book.

Closed books are sorted by `closedAt` descending (most recently closed first). If there are no closed books, the section is omitted entirely. If all books are closed, only the compact cards are shown.

## User Stories

1. As a user with many books, I want open books and closed books to be visually separated, so that I can quickly find my active books without scrolling past closed ones.

2. As a user, I want closed books to be displayed in a compact, space-efficient layout, so that the page doesn't feel cluttered with large cards for archived books.

3. As a user, I want to see the computed net balance of each closed book at a glance, so that I know what each book's balance was when it was closed.

4. As a user, I want closed books sorted by most recently closed first, so that the most relevant closed books appear at the top.

5. As a user, I want to still be able to edit a closed book's metadata (name, currency), so that I can correct or update archived book information.

6. As a user, I want the "New book" button to remain accessible below both sections, so that I can create a new book regardless of how many open or closed books exist.

7. As a user, I want to see a separator between open and closed books, so that the visual distinction is clear without requiring section headings.

8. As a user with only open books, I want the closed books section to be absent entirely, so that there is no unnecessary empty space or visual clutter.

9. As a user with only closed books, I want all books to render as compact cards, so that the page is still usable without an empty open-books section.

10. As a user, I want the balance for a closed book to display "—" while it is loading, so that the page is interactive immediately without waiting for all stats to resolve.

11. As a user, I want a failed balance fetch for one book not to block other books from displaying their balances, so that partial data is still visible.

12. As a developer, I want the closed-book balance logic encapsulated in a hook, so that it is testable in isolation and the component stays clean.

## Implementation Decisions

### Balance computation

Closed books do not have a `balance` field on `BookDto`. Instead, we call the existing `getBookStats(bookId)` endpoint for each closed book after the books list loads. This endpoint returns `{ transactionCount, totalSum }`. The `totalSum` value is displayed as the balance.

- All stats requests are batched with `Promise.all` to minimize load time.
- Balances display "—" while loading, then fill in with the formatted value.
- If a single book's stats call fails, the balance for that book stays at "—" (no error displayed, failure is silent).

### Balance display format

The balance uses `formatCurrency` from `@/lib/utils`, which formats the absolute value with configurable decimal places and wraps negative amounts in brackets (e.g., `1,250.00` or `(500.00)`). No currency symbol is displayed.

### Closed book card layout

Each closed book is rendered as a single-line compact card:

```
┌──────────────────────────────────────────────────────────┐
│ Vacation 2026  [Closed]                  1,250.00  [Edit] │
└──────────────────────────────────────────────────────────┘
```

- **Left**: Book name + "Closed" badge
- **Right**: Balance (formatted) + "Edit" button
- No "Select" button — closed books are not selectable
- Uses the same shadcn `Card` component but with compact padding

### Sorting

Closed books are sorted by `closedAt` descending (most recently closed first). The `closedAt` field is guaranteed to be present on closed books per the `BookDto` contract.

### Separator

A shadcn `<Separator />` component sits between the open and closed sections. No text headings are used.

### Edge cases

- **No closed books**: The separator and closed section are omitted entirely.
- **All books closed**: All books render as compact cards. No separator is shown (there is nothing to separate).
- **No books at all**: The existing empty state is preserved unchanged ("No books found. Create a book to get started.").

### Error handling

- **Initial page load failure**: Existing behavior unchanged — shows error with Retry button.
- **Balance fetch failure (individual book)**: The balance stays "—". No error toast or inline error. Silent failure.
- **Balance fetch failure (all books)**: All closed books show "—" for their balances.

### Modules to create and modify

| Module | Description | Type |
|--------|-------------|------|
| `features/books/hooks/useClosedBookBalances.ts` | Hook that accepts the full books list, filters closed books, fetches stats via `Promise.all`, sorts by `closedAt` desc, and returns a map of `{ [bookId]: totalSum \| null }` plus a loading flag | Create |
| `pages/book/BookPage.tsx` | Split rendering into open/closed sections, add Separator, render closed books as compact cards | Edit |
| `pages/book/BookPage.test.tsx` | Add tests for closed book rendering, sorting, balance display, separators, edge cases | Edit |

### Hook interface (from prototype)

```ts
interface ClosedBookBalance {
  bookId: string
  totalSum: number
}

interface UseClosedBookBalancesResult {
  /** Map from bookId to totalSum. null while loading or if fetch failed. */
  balances: Record<string, number | null>
  /** True while any stats request is in flight. */
  isLoading: boolean
}

function useClosedBookBalances(books: BookDto[]): UseClosedBookBalancesResult
```

- The hook calls `getBookStats` only for books with `status === "closed"`.
- If `books` has no closed entries, the hook returns immediately with an empty `balances` map and `isLoading: false`.
- The hook ignores stale responses using a counter pattern (matching the existing pattern in `useBooksStore.fetchBook`).

## Testing Decisions

Tests should verify behavior, not implementation details. For this feature:

- **`useClosedBookBalances.test.ts`** — Unit test the hook in isolation:
  - Returns empty map when no closed books exist
  - Fetches stats for each closed book
  - Batches requests with a single `Promise.all`
  - Handles partial failure (one book fails, others load)
  - Handles all failures (all balances remain null)
  - Sorts books correctly by `closedAt` descending
  - Ignores open books
  - Prior art: `useBooksStore.test.ts`, `useDailyInsight.test.ts`

- **`BookPage.test.tsx`** — Update the existing component tests:
  - Open books render as cards (existing behavior preserved)
  - Closed books render as compact single-row cards
  - A `<Separator />` renders between open and closed sections
  - No "Select" button on closed books
  - Closed books have an "Edit" button
  - Closed books display the balance value (or "—" while loading)
  - When there are no closed books, no separator is rendered
  - When there are only closed books, all render as compact cards (no separator)
  - "New book" button appears below all sections
  - Prior art: existing `BookPage.test.tsx`, `AddPage.test.tsx`, `HistoryPage.test.tsx`

## Out of Scope

- Adding a `balance` field to `BookDto` — balance is computed on-demand via `getBookStats`.
- Reopening a closed book from the BookPage — this is handled on the EditBookPage.
- Moving BookPage into `features/books/pages/` — folder restructuring is deferred to a separate task.
- Adding a "closed books count" or summary header.
- Pagination or virtualization of the closed books list.
- Sorting open books (order is determined by the API/service).
- Color-coding balances (positive/negative colors) — keep plain text.
- Caching closed book balances in the Zustand store — the component-level hook suffices.

## Further Notes

- The existing `getBookStats` endpoint already filters out `isBookClosingEntry` transactions, so the reported balance reflects the actual state of the book at close time, not including the balancing entry that was created for the Main book.
- The `formatCurrency` utility from `@/lib/utils` is used for consistency with the rest of the codebase (it's already used in insight pages, transaction rows, etc.).
- The compact closed-book card reuses the same `Card` component as open books but with reduced padding and a single-row layout, keeping visual consistency while distinguishing the two sections.
