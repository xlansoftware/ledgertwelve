# PRD 011 — Persist Current Book Selection

## Problem Statement

The current book selection exists only in client-side Zustand state (`useBooksStore.currentBook`). When a user reloads the page or opens the app in a new browser session, the selection is lost — it falls back to the first book in the list (`data[0]`). There is no mechanism to persist the user's preferred book across sessions or devices.

Additionally, the server does not guarantee a default "Main" book on user registration, which means some user flows rely on an implicit fallback rather than an explicit contract.

## Solution

Introduce a dedicated **Current Book** API resource (`/api/v1/books/current`) that persists the user's selected book on the backend. The server always creates a default "Main" book for every user at registration, guaranteeing that a `GET /api/v1/books/current` call always returns a valid book.

On login or app initialization, the frontend fetches the current book from the server instead of falling back to `data[0]`. When the user explicitly selects a different book, the frontend calls `PUT /api/v1/books/current` to persist the choice.

The frontend behaviour is unchanged otherwise — the store retains optimistic performance, and the server is the source of truth for the persisted selection.

## User Stories

1. As a user, I want my current book selection to persist across page refreshes, so that I don't have to re-select my book every time I reload the app.
2. As a user, I want my current book selection to persist across browser sessions, so that when I come back to the app later I land on the same book I was last working in.
3. As a user switching books, I want the new selection to persist immediately, so that if I reload the page right after switching, the new book is still selected.
4. As a new user with no explicit selection, I want the first book I own to be automatically selected, so that I never see an empty or error state for the current book.
5. As a new user who has never created a book, I want a default "Main" book to exist, so that the app always has a book to work with.
6. As a developer, I want the server to be the single source of truth for the current book selection, so that the frontend and backend contracts are clear and consistent.

## Implementation Decisions

### API Contract — New Endpoint

**`GET /api/v1/books/current`**

Returns the user's currently selected book. If the user has never explicitly selected a book, the server returns the first visible book ordered by creation date. The server guarantees that every user has at least one book (a default "Main" book created on registration).

Response (200):

```json
{
  "data": {
    "id": "book_main",
    "name": "Main",
    "currency": "EUR",
    "status": "open",
    "ownerId": "usr_1",
    "sharedWith": [],
    "createdAt": "2026-01-01T10:00:00Z"
  }
}
```

Errors: `401 Unauthorized`.

---

**`PUT /api/v1/books/current`**

Sets the user's current book. Validates that the book exists and is visible to the user.

Request:

```json
{
  "bookId": "book_vacation"
}
```

Response (200): Full `BookDto` of the newly selected book (same shape as GET).

Errors:
- `400` — `bookId` missing or invalid
- `404` — Book not found or not visible to the user
- `401` — Unauthorized

---

### DTO — No new types needed

The existing `BookDto` type is sufficient for both GET and PUT responses. No new DTOs are required.

### Server Guarantee

The backend must create a default "Main" book for every user on registration. This ensures:
- `GET /api/v1/books/current` never returns an empty/null result.
- The book close/reopen workflows, which reference the Main book, always find one.

### Modules Built / Modified

1. **`API.md`** — add `/api/v1/books/current` GET and PUT endpoint definitions. Add note about server creating a default "Main" book on registration.
2. **`src/services/booksService.ts`** — add `getCurrentBook()` and `setCurrentBook(bookId)` functions. Both are exported from `src/services/index.ts` via the existing `export * from "./booksService"`.
3. **`src/store/useBooksStore.ts`** — add `fetchCurrentBook()` and `setCurrentBook(bookId)` actions. In `fetchBooks()`, after loading the book list, also fetch the persisted current book via `fetchCurrentBook()` and use it instead of the `data[0]` fallback. The store remains the single place the rest of the app reads `currentBook` from.
4. **`src/mocks/handlers.ts`** — add MSW handlers for `GET /api/v1/books/current` and `PUT /api/v1/books/current`. The mock maintains an in-memory map of `userId → currentBookId`.
5. **`src/services/booksService.test.ts`** — add tests for `getCurrentBook()` and `setCurrentBook()`.

### Data Flow

```
App init / login
  → fetchBooks() (list)
  → fetchCurrentBook() (persisted selection)
  → useBooksStore.currentBook = result

User selects book on /books page
  → setCurrentBook(bookId) via store
  → PUT /api/v1/books/current
  → store updates currentBook optimistically
```

### Store Changes — `useBooksStore`

- Add `fetchCurrentBook` action: calls `getCurrentBook()` service and sets `currentBook`.
- Add `setCurrentBook(bookId)` action: calls `setCurrentBook(bookId)` service, updates `currentBook` with the returned `BookDto`.
- Modify `fetchBooks`: still loads the list, but after loading, calls `fetchCurrentBook` to get the persisted selection. Remove the `data[0]` fallback from `fetchBooks`.

### Fallback Behaviour

On first login (no persisted selection), the server returns the first visible book by creation date. This matches the existing frontend behaviour where `data[0]` is used as the fallback, so there is no user-visible change.

### Error Handling

- `GET /api/v1/books/current` failure: the store retains any previously cached `currentBook` value. If no cached value exists either, the store falls back to `data[0]` from the books list (same as today).
- `PUT /api/v1/books/current` failure: an error is set on the store, and the `currentBook` is reverted to its previous value. A toast/banner is left to the calling component.

## Testing Decisions

### Principles

- Test external behaviour, not implementation details.
- Use MSW for API mocking (already set up).
- Prefer user-visible assertions (element content, store state, API call results).

### What Makes a Good Test

- A test should mock the network boundary (MSW handlers), not the store internals.
- Service tests verify the HTTP request/response contract (correct URL, correct method, body serialisation, error handling).
- Store tests verify that the correct service is called and that state transitions happen properly.
- No tests should call real API endpoints.

### Test Coverage

1. **`booksService.test.ts`** — update existing test file:
   - `getCurrentBook`: returns a `BookDto` for the authenticated user.
   - `getCurrentBook`: returns the first book if no selection was persisted.
   - `getCurrentBook`: throws `401` on unauthenticated request.
   - `setCurrentBook`: sends `{ bookId }` in request body and returns the selected `BookDto`.
   - `setCurrentBook`: throws `404` for invalid/non-visible bookId.
   - Prior art: existing tests in `booksService.test.ts` (`getBooks`, `getBook`, `createBook`).

2. **`useBooksStore.test.ts`** — new test file:
   - `fetchCurrentBook`: sets `currentBook` in store on success.
   - `fetchCurrentBook`: keeps existing `currentBook` on failure.
   - `setCurrentBook`: updates `currentBook` optimistically on success.
   - `setCurrentBook`: reverts `currentBook` on API error.
   - Prior art: `useTransactionsStore.test.ts` for store testing patterns.

3. **No component/layout tests** — the change is purely in the data layer (service/store). No UI changes are introduced. If the existing `BookPage` tests rely on the `data[0]` fallback behaviour, they should be updated to account for the new flow.

## Out of Scope

- A user preferences/settings resource — if more user-level preferences emerge later (theme, default currency, default date range), they should get their own resource or a dedicated preferences endpoint. Current book is a book-related resource and lives under `/api/v1/books/current`.
- Session-scoped selection — the selection persists across sessions (server-side storage), not ephemeral cookies.
- Multi-device sync — the selection is per-user, so it naturally syncs across devices when the user logs in.
- UI changes to the book selector, header, or navigation — the existing components already read `currentBook` from the store and will work correctly once the store hydrates from the server.
- Backend schema or migration design — the PRD defines the API contract; the backend team implements the storage mechanism (e.g., a `CurrentBook` column on the `AspNetUsers` table, or a dedicated `UserPreferences` table).

## Further Notes

- The "Main" book creation on registration is a backend concern and is only documented here as a contract requirement. The frontend mocks already hard-code a "Main" book for test users, which satisfies the contract for development and testing.
- The MSW mock for current book selection uses a simple in-memory `Map<userId, bookId>` that resets between test runs but persists within a dev session — matching the behaviour of the future real backend.
- When the user has only one book (e.g., only "Main"), the `PUT` endpoint is never called because there is nothing to switch to. The `GET` endpoint simply returns that one book. This is fine and expected.
