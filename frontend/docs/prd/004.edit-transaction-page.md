# Edit Transaction Page

## Problem Statement

Users can view and delete transactions from the history page, but there is no way to edit a transaction's properties after it has been created. If a user makes a mistake (wrong amount, wrong category, wrong date, missing note) they have no recourse within the application.

## Solution

Introduce a dedicated edit transaction page at `/edit-transaction/:transactionId` that allows users to modify the properties of an existing transaction. The page is reached by clicking "Edit" in the transaction row menu on the history page. After saving or cancelling, the user returns to the history page.

## User Stories

1. As a user viewing the transaction history, I want to click an "Edit" option on a transaction row, so that I can navigate to a page where I can modify that transaction.
2. As a user on the edit transaction page, I want to see the current values of all editable fields pre-populated, so that I can see what I'm changing.
3. As a user on the edit transaction page, I want to modify the transaction's date, amount, currency details, category, and note, so that I can correct any mistakes.
4. As a user on the edit transaction page, I want to click "Save" to persist my changes, see a success toast, and return to the history page, so that I have immediate confirmation the edit worked.
5. As a user on the edit transaction page, I want to click "Cancel" to discard any changes and return to the history page, so that I can abort editing without side effects.
6. As a user on the edit transaction page, I want to see an inline error message if saving fails, and remain on the page, so that I can retry or fix the issue.
7. As a user who navigates directly to the edit page URL (e.g., bookmark or refresh), I want the transaction to be fetched and displayed, so that the page works as a deep link.
8. As a user on the edit transaction page, I want a loading state while the transaction is being fetched, so that I understand the page is not yet ready.

## Implementation Decisions

### Route

- New route: `/edit-transaction/:transactionId`
- Added to the existing router in `routes/index.tsx` as a child of `App`

### Navigation trigger

- `TransactionRowMenu.tsx`: The Edit menu item navigates via `useNavigate` to `/edit-transaction/${transaction.id}` instead of setting local state (the existing commented-out dialog code is removed).

### Data fetching

- On mount, the page calls `useTransactionsStore`'s `fetchTransaction(transactionId)` action.
- Categories are used from `useCategoriesStore`.
- A loading skeleton is shown while the fetch is in progress.
- If the transaction is not found (404), a "Transaction not found" error state is shown with a link back to history.

### Form fields

All fields are editable inline (no modal/dialog for multi-currency):

- **Date/time** — a date and time input
- **Amount** — numeric input (the book-currency amount)
- **Original currency** — text/select input for currency code
- **Original amount** — numeric input, shown when original currency is set
- **Exchange rate** — numeric input, shown when original currency is set
- **Category** — combobox showing categories names
- **Note** — text input

### Save flow

1. User clicks "Save"
2. `updateTransaction(transactionId, payload)` is called via the store
3. On success: `toast.success("Transaction updated")` → navigate to `/history`
4. On failure: inline error banner is displayed, form remains populated for retry

### Cancel flow

1. User clicks "Cancel"
2. Navigate to `/history` immediately (no dirty-state confirmation)

### Form layout

A conventional stacked form layout (not mirroring the AddPage layout). Fields are grouped logically:

- A page header with "Edit Transaction" title, Save and Cancel buttons
- Amount section (amount, original currency, original amount, exchange rate)
- Category picker
- Note input
- Date/time field

### Delete

- Not available on the edit page. Delete remains accessible only from the history page menu.

### Error handling

- Fetch errors: shown as an error state with a retry/"Go back" option
- Save errors: inline error banner at the top of the form, user stays on page to retry
- 404 on fetch: "Transaction not found" message with link to history

### State management

- The store's existing `fetchTransaction`, `updateTransaction`, and `clearCurrentTransaction` actions are used as-is. No new store actions are needed.

### API contract

- `GET /api/v1/transactions/{transactionId}` — already documented in API.md, already has MSW handler
- `PUT /api/v1/transactions/{transactionId}` — already documented in API.md, already has MSW handler
- `UpdateTransactionRequest` type in `services/transactionsService.ts` — already exists

### Modules

**New files:**
- `src/pages/edit-transaction/EditTransactionPage.tsx` — page component

**Modified files:**
- `src/pages/history/TransactionRowMenu.tsx` — replace local-state edit with navigation

**No changes needed:**
- `src/services/transactionsService.ts` — all required service methods exist

## Testing Decisions

### Testing philosophy
- Test external behavior (what the user sees and does), not implementation details
- Use MSW to mock API responses
- Prefer user-visible assertions (screen queries by role, label, test ID)
- Avoid testing internal state of the store or component

### Modules to test
- **`EditTransactionPage`** (`EditTransactionPage.test.tsx`):
  - Renders a loading skeleton while fetching
  - Pre-populates form fields with fetched transaction data
  - Save button calls update and navigates to /history on success
  - Cancel button navigates to /history
  - Shows error banner when save fails and stays on page
  - Shows "Transaction not found" when 404 is returned

### Prior art
- `HistoryPage.test.tsx` — uses MSW for mocking, `render` + `screen` queries, `fireEvent` for interactions, `waitFor` for async assertions
- `AddPage.test.tsx` — uses `vi.mock` for service mocking, `render` + `screen` + `fireEvent` + `waitFor`

### Test infrastructure
- MSW server is already set up via `@/test-setup`
- Tests will use the existing mock handlers (no new handlers needed)

## Out of Scope

- Deleting a transaction from the edit page (remains in history menu only)
- Changing the book/owner of a transaction
- Bulk editing transactions
- Undo functionality after save
- Audit trail of changes
- Mobile-responsive layout adaptations (should work within existing responsive framework)

## Further Notes

- The existing commented-out dialog in `TransactionRowMenu.tsx` (`{/* {edit && <EditTransactionForm ... />} */}`) should be removed entirely, replaced by the navigation approach.
- The `clearCurrentTransaction` action can be called on unmount of the edit page to clean up the store, but this is optional since the store's `currentTransaction` is only used by the edit page and doesn't affect the history page.
