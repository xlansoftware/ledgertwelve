# PRD: Offline / Local-Only Mode

## Problem Statement

The ledger12 app requires a live backend server for all operations. Users cannot use the app without network connectivity, and there is no way to demo or test the application without a running backend. The app needs a local-only mode where all data is stored entirely in the browser, enabling offline use, demos, and development without backend dependencies.

## Solution

Introduce a mode-switching architecture where the app can run in one of two modes:

- **Online mode** — all data operations call the existing backend API (current behavior, unchanged).
- **Offline mode** — all data operations read/write to IndexedDB in the browser, with no network calls.

The core abstraction is a set of per-domain service interfaces backed by a factory pattern. The factory returns either an online or offline implementation depending on the active mode. The mode is persisted in localStorage (`ledger12.mode`) and is selected by the user from the settings page or a login-screen link.

## User Stories

1. As a user, I want to use the app without an internet connection, so that I can track my finances anywhere.
2. As a user, I want all my data persisted across page refreshes in offline mode, so that I don't lose my entries.
3. As a user, I want to create, edit, and delete books while offline, so that I can manage my accounts.
4. As a user, I want to create, edit, and delete categories while offline, so that I can organize my spending.
5. As a user, I want to create, edit, and delete transactions while offline, so that I can record my expenses.
6. As a user, I want to view paginated transaction history with date-range and category filters while offline, so that I can review past entries.
7. As a user, I want to view daily and monthly reports computed from my local data, so that I can analyze my spending patterns.
8. As a user, I want to view book statistics (transaction count, net sum) while offline, so that I can understand my account balances.
9. As a user, I want to close and reopen books while offline, so that I can manage book lifecycles locally.
10. As a user, I want to switch between online mode and offline mode from the settings page, so that I can go back to the backend when connectivity is available.
11. As a user, I want the app to remember my mode choice across sessions, so that I don't have to re-select it every time.
12. As a developer, I want to run the app entirely offline without MSW or a backend, so that I can develop and demo the app quickly.
13. As a user in offline mode, I want the UI to hide or disable features that don't work locally (sharing, exports, exchange rates), so that I'm not confused by non-functional buttons.
14. As a developer, I want each domain's service interface to be testable in isolation, so that offline implementations can be verified without a browser or backend.

## Implementation Decisions

### Architecture

- **Per-domain interfaces**: Five TypeScript interfaces (`IBooksService`, `ICategoriesService`, `ITransactionsService`, `IReportsService`, `IUsersService`) each mirroring the method signatures of the existing `src/services/*.ts` modules method-for-method.
- **Factory pattern**: A service-locator-style singleton (`setFactory()` / `getFactory()`) holds the current `ServiceFactory` (an object with all five service instances). Created during app bootstrap based on the persisted mode.
- **Mode switching**: Writing `"online"` or `"offline"` to localStorage key `ledger12.mode` followed by `window.location.reload()`. No hot swapping.
- **Existing code untouched**: `src/services/*.ts` files remain unchanged. Online wrappers delegate to them. Stores call `getFactory().transactions.getTransactions()` instead of importing services directly.

### Domains covered by the factory

| Domain | Included | Notes |
|--------|----------|-------|
| Books | Yes | Full CRUD, close/reopen, sharing, stats |
| Categories | Yes | Full CRUD, reassign, reorder |
| Transactions | Yes | Full CRUD, paginated list with filters |
| Reports | Yes | Daily, monthly, totals, category breakdown, averages |
| Users | Yes | Returns the single local user |
| Auth | No | Handled separately via existing `setLocalMode()` |
| Exchange rates | No | Not available in offline mode |
| Exports | No | Not meaningful in local-only context |

### IndexedDB schema

- **Database name**: `"ledger12"`, versioned from day one (v1).
- **Object stores**:
  - `books` — keyed by `id` (UUID). Stores `BookDto` documents.
  - `categories` — keyed by `id` (UUID). Stores `CategoryDto` documents.
  - `transactions` — keyed by `id` (UUID). Stores `TransactionDto` documents.
    - Indexes: `bookId` (standalone), compound `[bookId, dateTime]`.
    - Additional filter indexes (category, userId) deferred — filter in memory.
  - `users` — keyed by `id` (UUID). Stores `UserSummary` documents.
  - `sharedUsers` — keyed by compound `[bookId, userId]`. Stores `{ bookId, userId, email, permission }`.
- **Migration**: `onupgradeneeded` handler creates stores and indexes per version.

### ID generation

UUIDs via `crypto.randomUUID()`. No prefix convention. Generated client-side for all new entities in offline mode.

### Offline user identity

A UUID is generated via `crypto.randomUUID()` on first offline launch and persisted in localStorage. The `OfflineUsersService` returns this user with email `"local@ledger12.app"`.

### Report computation

All report endpoints recompute from scratch on every call. The `OfflineReportsService` pulls the relevant transactions from IndexedDB by date range, groups/sums/averages in memory, and returns the same DTO shapes as the server endpoints.

### Book stats

Computed on the fly from stored transactions — no precomputed cache.

### Close book workflow (offline)

`OfflineBooksService.closeBook()`:
1. Computes net balance by fetching all non-closing transactions for the book from IndexedDB.
2. Calls `getFactory().transactions.createTransaction()` to create a closing entry in the Main book.
3. Updates the book's status to `"closed"` in IndexedDB.

Offline services receive the full factory as a constructor argument to enable cross-service workflows.

### Sharing in offline mode

Sharing methods (`addShare`, `updateShare`, `removeShare`) have real implementations that read/write the `sharedUsers` store in IndexedDB. The data is stored locally for completeness and future upload scenarios, even though sharing has no effect in offline mode.

### Bootstrap flow

1. `main.tsx` reads `ledger12.mode` from localStorage.
2. If `"offline"`: skip MSW startup, skip `whoami()`, create offline factory, set auth store to `{ status: "local" }`, call `initializeApp()` (which now goes through the factory), render.
3. If `"online"` (or absent): same flow as today — start MSW, check session, create online factory, call `initializeApp()`, render.
4. If session check fails: show login page with an "Offline mode" link.

### Modified existing modules

- **`src/store/useAuthStore.ts`** — no interface changes, but `setLocalMode()` will also persist `ledger12.mode` to localStorage and reload.
- **`src/main.tsx`** — adds the offline bootstrap branch before the MSW/session check.
- **`src/store/useBooksStore.ts`** — replace direct service imports with `getFactory().books.*`.
- **`src/store/useCategoriesStore.ts`** — same pattern.
- **`src/store/useTransactionsStore.ts`** — same pattern.
- **`src/store/useUsersStore.ts`** — same pattern.
- **`src/lib/init.ts`** — no change needed (it calls stores, which go through the factory).
- **Settings page** — add a toggle between online and offline mode that writes to localStorage and reloads.
- **Login page** — add an "Offline mode" link that triggers the switch.
- **Route guards / UI components** — conditionally hide sharing UI, export buttons, and rate tools when in local mode.

## Testing Decisions

### Test philosophy

- Test behavior, not implementation details.
- Prefer user-visible assertions.
- Mock network requests using MSW in online tests.
- In offline tests, seed IndexedDB with known data and assert on query results.
- Avoid testing internal IndexedDB call patterns — test the service interface methods.

### Modules to test

1. **`OfflineTransactionsService`** — highest complexity. CRUD cycle, paginated list, date-range filtering, in-memory filter fallback (category, note, value). Seed transactions, call service methods, assert returned DTOs match expectations.

2. **`OfflineReportsService`** — computation logic. Seed known transactions into IndexedDB, call each report method, assert correct grouping, summing, averaging, and date filtering.

3. **`OfflineBooksService`** — CRUD cycle, close/reopen workflow (verify balancing transaction is created), sharing operations, stats computation.

4. **`OfflineCategoriesService`** — CRUD cycle, reassign (verify affected transactions update), reorder.

5. **`db.ts`** — schema creation: verify all object stores and indexes exist after `openDb()` resolves.

6. **`OfflineUserStore`** — first-launch UUID generation, persistence across simulated page refreshes.

7. **Existing store tests** — update to use `setFactory()` with a mock factory instead of direct service imports. No behavioral changes expected.

### Prior art

- Existing tests in `src/services/*.test.ts` and `src/store/*.test.ts` provide patterns for async service testing and store mocking.
- Use Vitest with `beforeEach`/`afterEach` to seed and clear IndexedDB.
- Use `fakeIndexedDB` from `fake-indexeddb` npm package for offline service tests (no real browser IndexedDB needed).

## Out of Scope

- **Sync/upload**: Uploading offline data to the backend is not implemented. The `sharedUsers` store is structured for future sync but no sync mechanism exists.
- **Exchange rates**: No exchange rate lookups in offline mode. Multi-currency transactions must use the same currency or assume 1:1.
- **Exports**: No XLSX/CSV export generation in offline mode.
- **Authentication in offline mode**: No login screen, no password, no session management. The user is always the single local user.
- **Conflict resolution**: If the user switches modes frequently, no attempt is made to merge data between IndexedDB and server.
- **PWA / service worker**: Offline mode is a data-layer abstraction, not a full progressive-web-app with offline-first networking and service worker caching.
- **IndexedDB performance optimization**: In-memory filtering after compound-index scan is accepted for local data volumes. No query planner, no perf benchmarks.

## Further Notes

- The mode switch triggers a full page reload. This is a deliberate simplicity trade-off — the loading spinner already exists in the bootstrap flow and provides clear visual feedback.
- Sharing methods have real offline implementations even though sharing is non-functional offline. This preserves the interface contract and prepares for a future upload feature.
- The `sharedUsers` IndexedDB store is normalized (separate from the `books` document) to support future upload scenarios where shares need to be reconciled independently.
- All new code lives in `src/features/offline/` following the feature-first organization rule in AGENTS.md.
